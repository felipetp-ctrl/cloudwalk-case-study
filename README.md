# ML-Based Anomaly Detection for Edge HTTP Traffic

[![CI](https://github.com/felipetp-ctrl/cloudwalk-case-study/actions/workflows/ci.yml/badge.svg)](https://github.com/felipetp-ctrl/cloudwalk-case-study/actions/workflows/ci.yml)

An ML pipeline that classifies incoming HTTP request sequences as malicious or benign in real time at the edge, enabling automated blocking before requests reach origin servers. Built for CloudWalk's edge infrastructure constraints: <5ms inference, WASM runtime, no GPU.

See **[WRITEUP.md](WRITEUP.md)** for the full technical writeup covering reasoning, results, and deep-dive answers.

## Prerequisites

- **Python 3.10+**
- **Rust toolchain** — only needed for the edge-inference benchmark (`rustup` — https://rustup.rs)
- **wasmtime** — only needed for the WASM benchmark (`brew install wasmtime` on macOS)

## Running

### 1. Python Setup

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Register the kernel so Jupyter can find it:

```bash
python -m ipykernel install --user --name cloudwalk --display-name "CloudWalk"
```

### 2. Notebooks (run in order)

Each notebook depends on the output of the previous one. Run them sequentially:

```bash
jupyter notebook
```

| Order | Notebook | What it does | What it produces |
|---|---|---|---|
| 1 | `notebooks/eda.ipynb` | Joins incident labels to requests (IP, CIDR, TLS + temporal bounds). Exploratory analysis of attack patterns. | Labeled dataset with 592 malicious / 49,408 benign requests |
| 2 | `notebooks/feature_engineering.ipynb` | Computes 15 per-request + 21 IP session features with causal windowing. Analyzes feature distributions. | Feature matrix ready for modeling |
| 3 | `notebooks/baseline_model.ipynb` | Trains 4 models (LR, RF, XGB, LGBM) with Optuna tuning. Evaluates on temporal split. Pruning and threshold analysis. | `plots/pr_curves.png`, `plots/feature_importance.png`, `plots/cost_curve.png`, trained LGBM model |
| 4 | `notebooks/edge_deployment.ipynb` | Exports both tier models to ONNX. Benchmarks Python inference (LightGBM native vs ONNX Runtime). Prepares models for Rust/WASM benchmark. | `edge-inference/model/model_full.onnx`, `edge-inference/model/source_model.onnx` |

### 3. Tests

Run the full test suite from the project root:

```bash
pytest -v
```

Expected output: **63 tests passed**.

Individual test modules:

```bash
pytest tests/test_label_joining.py -v   # 4 tests — label joining strategies

pytest tests/test_model.py -v           # 21 tests — training, evaluation, tuning, pruning
pytest tests/test_source_model.py -v   # 9 tests — source-level features, training, ensemble
pytest tests/test_export.py -v          # 3 tests — ONNX export, validation, source model export
pytest tests/test_pipeline.py -v        # 2 tests — end-to-end pipeline
pytest tests/test_monitoring.py -v      # 14 tests — drift, PSI, predictions, performance
```

### 4. Edge Inference Benchmark (Rust/WASM)

Requires notebook 4 to have run first (generates the ONNX model).

```bash
cd edge-inference
```

**Native Rust benchmark:**

```bash
rustup target add wasm32-wasip1    # one-time setup
cargo build --release
cargo run --release
```

**WASM benchmark:**

```bash
cargo build --release --target wasm32-wasip1
wasmtime run --dir model target/wasm32-wasip1/release/edge-inference.wasm
```

Both benchmarks report p50/p95/p99 latency and throughput per thread. Expected results: p99 < 1ms for both native and WASM (well within the 5ms budget).

### 5. Using the Pipeline Programmatically


## AI-Assisted Development: Ethical and Socratic Approach

This project was developed using AI as a collaborative thinking partner, following principles of **ethical AI use** and the **Socratic method** for learning and problem-solving.

### How AI Was Used

**Thinking Partner, Not Replacer:**
- AI assisted in **exploratory analysis** — asking clarifying questions about anomaly detection strategies, edge constraints, and trade-offs before implementing
- **Architecture validation** — testing design decisions against known patterns and edge cases (latency budgets, model complexity, deployment targets)
- **Code review and reasoning** — questioning implementation choices, suggesting improvements, and validating correctness
- **Documentation** — translating technical decisions into clear narratives for future maintainers

**Not Used For:**
- Blindly copying solutions without understanding
- Generating untested code without verification
- Making business or security decisions unilaterally
- Replacing human judgment on trade-offs (model accuracy vs. latency, feature engineering choices, etc.)

### Ethical Principles Applied

1. **Transparency**: Every decision is documented with reasoning in `WRITEUP.md` and `docs/` — you can trace *why* we chose LightGBM, pruning thresholds, edge targets, etc.

2. **Verification**: All generated code is tested (63 unit tests) and validated against real constraints (5ms latency, WASM runtime, no GPU).

3. **Accountability**: Code review and testing ensure quality; AI suggestions are always questioned before acceptance.

4. **Human Judgment**: Trade-offs (false negative cost, feature complexity, deployment risk) remain human decisions, informed by AI analysis but not dictated by it.

5. **Bias Awareness**: Dataset is synthetic (known limitations documented), feature engineering is explicit (no black-box automatic selection), and model pruning is guided by security priorities, not just accuracy metrics.

Every notebook cell includes reasoning about *why* a step was taken, not just *what* it does. The WRITEUP documents the journey — false starts, rejected approaches, and the reasoning that led to final decisions.
