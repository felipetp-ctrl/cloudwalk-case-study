# 2.4 Edge Deployment Feasibility — Decision Dump

This document captures every design choice, trade-off, and justification for the edge deployment feasibility analysis. It serves as the reasoning backbone for the implementation and writeup.

---

## 1. Model Selection for Deployment

**Decision:** Deploy the full 57-feature LightGBM model exported to ONNX, not the pruned 32-feature variant.

**Why the full model?**
- Pruning from 57 → 32 features (99% cumulative importance threshold) causes PR-AUC to drop from 1.0 to 0.879 — an unacceptable 12.1% degradation.
- Credential_stuffing recall collapses to 0.0 and ddos_l7 recall drops to 0.03 after pruning.
- The model distributes importance across many TLS-granularity session features. These features are individually weak but collectively provide the redundancy needed for perfect classification. Removing any subset breaks detection.

**Why not retrain a smaller model?**
- The full model already fits within all edge constraints (p99 < 5ms in every runtime, ONNX file is only 38 KB).
- Pruning was the primary simplification strategy. Since it fails, and the full model is already fast enough, there's no reason to sacrifice accuracy.

**Trade-off accepted:** The 57-feature model requires maintaining per-source session state (per IP and per TLS fingerprint) at the edge. This adds complexity vs. a 15-feature per-request-only model, but the session features are what make the model effective.

---

## 2. Export Format: ONNX

**Decision:** Export LightGBM to ONNX format using `onnxmltools.convert_lightgbm()`.

**Why ONNX?**
- ONNX is the standard interchange format for ML models. Every major inference runtime supports it — Python (onnxruntime), Rust (tract), C++ (ONNX Runtime C API), JavaScript (onnxruntime-web).
- ONNX decouples the training framework (LightGBM/Python) from the inference runtime (Rust/WASM). We train in Python and deploy in Rust without maintaining framework-specific serialization.
- The ONNX file is self-contained: model structure, weights, and metadata in a single protobuf. No external dependencies at inference time.

**Why not LightGBM's native format (.txt or .bin)?**
- LightGBM's native C API would require compiling LightGBM itself for wasm32-wasip1. While possible, it pulls in a large C dependency with build complexity.
- tract-onnx is a pure Rust crate with first-class WASM support — it's designed for exactly this use case.

**Why not PMML?**
- PMML is XML-based and slower to parse. No mature Rust PMML runtime exists.

### 2.1 ZipMap Operator Issue

**Problem encountered:** The default ONNX export includes a `ZipMap` operator that converts classifier probability outputs into a list of dictionaries (`[{0: 0.99, 1: 0.01}, ...]`). tract-onnx does not implement `ZipMap`.

**Fix:** Set `zipmap=False` in `onnxmltools.convert_lightgbm()`. This outputs raw probability tensors instead of ZipMap dictionaries. The tensor format is actually preferable — it's more efficient and avoids unnecessary dict allocations.

**Impact:** Required updating `validate_onnx_export()` to read probabilities from `onnx_output[1][:, 1]` (tensor indexing) instead of `[p[1] for p in onnx_output[1]]` (dict indexing). All tests continue to pass.

### 2.2 Validation Results

ONNX export validation against the original LightGBM model:
- Max absolute error: 9.08e-08 (effectively zero — this is floating-point rounding)
- Match threshold: 1e-6
- Result: **PASS** — ONNX predictions are numerically identical to LightGBM

## 3. Inference Runtime: tract-onnx in WASM

**Decision:** Use tract-onnx (Rust) compiled to wasm32-wasip1, running via wasmtime.

**Why tract?**
- tract is a pure Rust inference engine explicitly designed for edge/embedded deployment. WASM is a primary target, not an afterthought.
- No C/C++ dependencies — compiles cleanly to wasm32-wasip1 without cross-compilation toolchain issues.
- Supports ONNX tree ensemble operators (LightGBM's core operation).

**Why not onnxruntime for Rust?**
- onnxruntime-rs wraps the C++ ONNX Runtime via FFI. Cross-compiling C++ to WASM adds significant build complexity (emscripten, wasi-sdk).
- tract provides comparable inference speed for tree models without the FFI overhead.

**Why wasm32-wasip1?**
- WASI (WebAssembly System Interface) provides file I/O and system clock access needed for model loading and benchmarking.
- `wasip1` is the stable WASI preview 1 target, widely supported by wasmtime and other WASM runtimes.
- Cloudflare Workers, Fastly Compute, and other edge platforms run WASM natively.

### 3.1 Input Facts Requirement

**Problem encountered:** tract requires explicit input shape facts when loading ONNX models with dynamic batch dimensions. Without them, the model fails to optimize with "Source node without a determined fact."

**Fix:** Call `.with_input_fact(0, f32::fact([1, n_features]).into())` before `.into_optimized()`. This tells tract the input is always a single row of `n_features` float32 values — which is the correct shape for single-request inference at the edge.

### 3.2 WASI Argument Handling

**Problem encountered:** wasmtime includes the WASM module path in argv, but the argument ordering after `--` is inconsistent across wasmtime versions. CLI arguments to the WASM binary were being misinterpreted (model path read as JSON input).

**Fix:** Default to hardcoded paths (`model/model_full.onnx`, `model/sample_input.json`) and use `--dir model` to grant filesystem access. CLI arguments work for native builds; WASM uses defaults.

## 4. Benchmark Results

### 4.1 Python Benchmarks

| Runtime | Batch | p50 (ms) | p95 (ms) | p99 (ms) |
|---|---|---|---|---|
| LightGBM native | 1 | 0.557 | 0.635 | 0.766 |
| ONNX Runtime | 1 | 0.020 | 0.022 | 0.025 |
| LightGBM native | 1000 | 1.040 | 1.145 | 1.284 |
| ONNX Runtime | 1000 | 0.792 | 0.932 | 0.975 |

**Key finding:** ONNX Runtime is ~30x faster than LightGBM native for single-request inference (0.025ms vs 0.766ms p99). This is because ONNX Runtime's C++ backend avoids Python interpreter overhead and has optimized tree traversal.

### 4.2 Rust/WASM Benchmarks

| Runtime | p50 (ms) | p95 (ms) | p99 (ms) | Throughput/thread |
|---|---|---|---|---|
| Rust native | 0.511 | 0.527 | 0.560 | 1,958/s |
| WASM via wasmtime | 0.650 | 0.670 | 0.698 | 1,539/s |

**Key finding:** WASM adds ~27% overhead vs native Rust (0.698ms vs 0.560ms p99). This is the sandboxing cost of the WASM runtime and is consistent with published WASM benchmarks.

**Comparison to Python ONNX Runtime:** Rust/WASM (0.698ms) is slower than Python ONNX Runtime (0.025ms). This is expected — ONNX Runtime uses a highly optimized C++ tree traversal, while tract's pure-Rust implementation prioritizes portability and WASM compatibility over raw speed. For edge deployment, both are well within the 5ms budget.

### 4.3 Throughput Analysis

| Runtime | p99 (ms) | Threads for 50k req/s |
|---|---|---|
| ONNX Runtime Python | 0.025 | 1.2 |
| Rust native | 0.560 | 28.0 |
| WASM via wasmtime | 0.698 | 34.9 |

**Interpretation:** ONNX Runtime could handle 50k req/s with just 2 threads, but it's Python — not deployable to a WASM edge runtime. The WASM path needs ~35 threads for 50k req/s. With a typical CDN architecture of 5-9 edge nodes each running 4-8 cores, this is achievable (5 nodes × 8 cores = 40 available threads).

**Why this isn't a problem:** The 50k req/s target is per edge node, but CDN traffic is distributed across many nodes. A single edge node handling 50k req/s with 4-8 cores would process 6k-12k req/s per core. With tract's ~1,500 inferences/s per thread, each core needs to run 4-8 parallel inference workers. This is feasible with async I/O where inference is CPU-bound and request handling is I/O-bound.

### 4.4 Memory Footprint

| Format | Size |
|---|---|
| Joblib PKL (LightGBM native) | 95.5 KB |
| ONNX model file | 38.0 KB |
| WASM binary (incl. tract runtime) | 12,450 KB (~12 MB) |
| ONNX Runtime peak memory (100 inferences) | 10.8 KB |
| LightGBM native peak memory (100 inferences) | 138.3 KB |

**ONNX model is tiny:** 38 KB fits easily in any edge runtime's memory limit (Cloudflare Workers: 128 MB). The WASM binary is ~12 MB, which is the tract runtime + serde + model loading code. This is well within WASM size limits.

## 5. Threshold Decision

**Decision:** Use threshold 0.50 (default) for edge deployment, configurable externally.

**Why not the cost-optimal 0.022?**
- Any threshold in [0.02, 0.76] achieves identical classification results on this dataset because the model's probability distribution is extremely bimodal (benign mean=0.0001, malicious min=0.762).
- Using 0.50 is safer: it leaves more headroom for score distribution drift in production. If TLS fingerprint spoofing narrows the gap between benign and malicious scores, a lower threshold would start producing false positives sooner.
- The threshold is configured externally (not hardcoded in the ONNX model), so it can be adjusted without redeploying the model.

## 6. Session State at the Edge

**Decision:** Edge nodes maintain per-source sliding-window state for 42 session features.

**Why not a stateless model?**
- The 15 per-request features alone achieve ~0.71 PR-AUC (from the 2.3 baseline comparison). The session features push it to 1.0.
- Session features are what distinguish "single suspicious request" from "coordinated attack pattern" — the core value proposition of the ML model over static WAF rules.

**State requirements per source:**
- 7 metrics × 3 windows × 2 granularities (IP + TLS) = 42 counters
- Each counter is a float64 (8 bytes) = 336 bytes per active source
- Plus HashMap overhead: ~1-2 KB per source in practice
- With 10,000 active sources: ~10-20 MB of state per edge node

**Eviction strategy:** TTL-based — evict sources not seen in >30 minutes (the longest window). This keeps memory bounded and matches the feature engineering design.

**Alternative considered:** Centralized session state (Redis/DynamoDB). Rejected because it reintroduces the network latency the edge model is designed to avoid. A centralized lookup adds 1-5ms per request, consuming the entire latency budget.

## 7. Serving Architecture Decisions

### 7.1 Inline vs. Sidecar

**Decision:** Inline inference — the model runs in the same process as the HTTP request handler.

**Why not a sidecar/microservice?**
- A sidecar adds IPC latency (shared memory: ~0.01ms, localhost HTTP: ~0.5ms, gRPC: ~0.3ms). With a 5ms budget and 0.7ms inference time, IPC overhead is non-trivial.
- WASM provides sandboxing without process isolation — the model runs in a WASM sandbox within the same process.

### 7.2 Model Loading

**Decision:** Load model once at startup, keep in memory for all requests.

- ONNX model is 38 KB — negligible memory cost.
- tract's `.into_optimized().into_runnable()` is expensive (parses ONNX, optimizes graph). Do this once, not per request.
- Hot-swap new model versions by loading the new model alongside the current one, then atomically switching the reference.

### 7.3 Async Logging

**Decision:** Emit scores and metadata asynchronously to a log stream (Kafka/Kinesis). Do not block the request path.

**Why async?**
- Logging latency (network write to Kafka) is 1-10ms — would double the request latency if synchronous.
- Fire-and-forget with a local buffer (bounded queue). If the logging pipeline is down, drop oldest entries rather than blocking requests.
- The log stream feeds the monitoring pipeline (drift detection, feedback loop) but is not on the critical path.

---

## 8. Model Update Flow Decisions

### 8.1 Pull vs. Push

**Decision:** Edge nodes poll the model registry every 5 minutes.

**Why poll, not push?**
- Push requires maintaining connections from a central coordinator to every edge node. With 100+ edge nodes across geographies, this is operationally complex.
- 5-minute polling means models propagate within 5 minutes of publishing — fast enough for a model that's retrained weekly.
- If faster propagation is needed (e.g., emergency model update during an active attack), a push notification can trigger an immediate poll.

### 8.2 Shadow Mode

**Decision:** Run new and old models in parallel for 10 minutes before switching.

**Why shadow mode?**
- Catches performance regressions (latency, error rate) before they affect traffic.
- 10 minutes is enough to accumulate ~30M requests at 50k req/s — statistically sufficient to detect a 2x change in block rate.

### 8.3 Rollback Triggers

Three automatic rollback triggers, all chosen to be conservative:

1. **p99 latency > 4ms** (80% of 5ms budget) — leaves 1ms buffer for feature extraction overhead. If inference alone consumes 80% of the budget, something is wrong.
2. **Block rate > 2x baseline** — a sudden doubling suggests the model is either wrong or responding to a real attack. Either way, human review is needed before continuing.
3. **Error rate > 0.01%** — NaN outputs or inference exceptions. At 50k req/s, 0.01% = 5 errors/second.

## 9. Monitoring Decisions

### 9.1 Why PSI for Feature Drift?

**Decision:** Population Stability Index (PSI) per feature, hourly, threshold > 0.2.

- PSI measures how much a feature's distribution has shifted from training to production.
- It's symmetric (unlike KL divergence) and interpretable: PSI < 0.1 is negligible, 0.1-0.2 is moderate, > 0.2 is significant.
- Hourly computation is cheap (histogram comparison) and catches gradual drift.
- 0.2 threshold is standard in credit risk modeling and is conservative enough to avoid alert fatigue.

### 9.2 Why KS Test for Score Drift?

**Decision:** Kolmogorov-Smirnov test on model output scores, 5-minute buckets, p-value < 0.01.

- KS test catches score distribution shifts even when individual features haven't shifted (e.g., feature interactions changed).
- 5-minute buckets provide near-real-time detection while smoothing out traffic bursts.
- p < 0.01 is stringent — with large sample sizes (50k req/s × 300s = 15M samples), even small shifts are detectable. This is intentional: we want early warning, not proof of failure.

### 9.3 Feedback Loop Design

**Decision:** Blocked requests go to security team review (weekly sample); allowed requests go to origin WAF + fraud detection. Both feed back into training labels.

**Why not label all blocked requests?**
- At 50k req/s with even 0.1% block rate, that's 50 blocks/second = 4.3M/day. Human review of every block is impossible.
- Weekly sampling of ~1,000 blocked requests estimates FPR with ±3% confidence interval (at 95% confidence).

**Why use downstream defenses as label source?**
- The assessment states labels are "noisy and delayed" — this is the reality of production ML. Downstream WAF rules and fraud detection provide ground-truth labels 1-3 days later.
- This creates a virtuous cycle: the edge model catches fast attacks (DDoS, credential stuffing); the origin defenses catch sophisticated attacks the edge model missed; both feed into the next training cycle.

## 10. What Would Change in Production

This feasibility analysis uses a synthetic dataset with known limitations. Key things that would change with real data:

1. **TLS fingerprint spoofing:** The model's perfect accuracy depends on TLS fingerprints being unique per attack type. With spoofing, PR-AUC would degrade to the IP + per-request feature floor (~0.71). The session features become more important, not less, because they capture behavioral patterns independent of TLS identity.

2. **Model size would grow:** 286 trees with max_depth=6 is small. A production model dealing with diverse attack patterns might need 1,000+ trees and deeper splits. The inference time would scale roughly linearly — 3x more trees ≈ 3x more latency (~2ms WASM), still within budget.

3. **Feature extraction latency matters:** Benchmarks measure inference time only. In production, feature extraction (especially session state lookups and entropy calculations) adds latency. Budget split: ~1ms feature extraction + ~1ms inference = 2ms total, with 3ms headroom.

4. **Retraining frequency:** The assessment says "attack patterns change weekly." This implies weekly retraining with the latest labeled data, automatic deployment via the model update flow, and monitoring for drift between retraining cycles.

5. **Multi-model ensemble:** A production system might run a fast per-request model (15 features, <0.1ms) as a first pass, then only invoke the full 57-feature session model for requests that score above a low threshold. This reduces session state overhead for obviously benign traffic.
