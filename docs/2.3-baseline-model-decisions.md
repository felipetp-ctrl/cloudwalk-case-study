# 2.3 Baseline Model — Decision Dump

This document captures every design choice, trade-off, and justification for the baseline model pipeline. It serves as the reasoning backbone for the implementation and writeup.

---

## 1. Model Selection

**Decision:** Train and compare four models in a simple → complex progression: Logistic Regression → Random Forest → XGBoost → LightGBM.

**Why four models?**
- LR establishes a linear lower bound. If trees barely beat LR, the non-linear signal is weak and the simpler model wins for edge deployment.
- RF is a bagging baseline — shows whether boosting's sequential error correction actually matters or if parallel trees suffice.
- XGBoost and LightGBM are the real contenders. Comparing both lets us pick the one with the best accuracy-to-size ratio for WASM export.

**Why not deep learning?**
- WASM runtime, no GPU, <5ms latency budget. LSTMs and transformers are out.
- Dawadi et al. (2023) achieved 97.57% DDoS detection with LSTM but reports no latency benchmarks — inference requires significant compute.
- Tree models achieve comparable performance on tabular data with microsecond-level inference (Mayhew et al., 2015 reports SVM classification "in the microsecond range").

**Why not SVM?**
- SVMs scale poorly with dataset size (O(n²) to O(n³) training complexity). With 50k samples and Optuna tuning, training time becomes prohibitive.
- LightGBM/XGBoost match or exceed SVM accuracy on tabular data with faster training.
- SVM lacks native feature importance — harder to do the pruning analysis.

**Why not Isolation Forest?**
- Ness et al. (2025) found Isolation Forest achieved only 0.4 test accuracy on KDDCup'99, far below supervised methods.
- We have labels — using unsupervised methods wastes information.

**Academic support:**
- Ness et al. (2025, IEEE Access): Compared Isolation Forest, Naive Bayes, XGBoost, LightGBM, SVM on KDDCup'99. LightGBM achieved training accuracy 1.0 and test accuracy 0.85. XGBoost: 0.99/0.83.
- Kamol et al. (2024, ICCIT): Used DBSCAN clustering on web access logs for credential stuffing — achieved 90.56% accuracy. Validates IP-level aggregation but unsupervised only.
- Dawadi et al. (2023, Sensors): LSTM-based WAF with layered architecture. 97.57% DDoS accuracy. Too heavy for our constraints — serves as contrast justification for trees.
- Mayhew et al. (2015, MILCOM): BBAC system using SVM + clustering for HTTP traffic behavioral analysis. Pipeline mirrors ours: feature extraction → enrichment → classification. Real-time latency ~10ms.

## 2. Split Strategy

**Decision:** Temporal train/test split (train days 6-9, test days 10-12) + expanding-window temporal CV for Optuna tuning. No future data leaks into training.

### 2.1 Temporal Split Design

| Day | Attack Types | Split |
|---|---|---|
| Jan 6 | None (0 malicious) | Train |
| Jan 7 | credential_stuffing (111), scanner (25) | Train |
| Jan 8 | api_abuse (34) | Train |
| Jan 9 | api_abuse (31), ddos_l7 (146) | Train |
| Jan 10 | zero_day_exploit (6) | Test |
| Jan 11 | credential_stuffing (138) | Test |
| Jan 12 | ddos_l7 (101) | Test |

**Cutoff: Jan 10.** Train sees 4 attack types (credential_stuffing, scanner, api_abuse, ddos_l7). Test sees 3 types (zero_day_exploit, credential_stuffing, ddos_l7). This is realistic: in production, the model encounters future traffic that may include novel attack types never seen during training.

**Key properties:**
- `zero_day_exploit` appears only in test — the model has never seen this attack type. Without TLS fingerprint shortcuts (removed due to circular dependency — see 2.2 section 11), the model has no way to detect a novel attack type with only 6 samples never seen in training. The 0% recall on zero_day_exploit is expected and honest. The unsupervised/hybrid strategy discussed in assessment 3.2 remains the robust approach for truly novel attacks.
- `scanner` and `api_abuse` appear only in train — tests whether the model overfits to specific attack type signals.
- `credential_stuffing` and `ddos_l7` appear in both — allows evaluation of generalization across time for known attack types.

### 2.2 Split Layout

| Usage | Method | Purpose |
|---|---|---|
| Train/Test | Temporal cutoff at Jan 10 (~58/42) | Final evaluation |
| CV Folds 1-3 | Expanding window (6→7, 6-7→8, 6-8→9) | Optuna tuning |

**Trade-off:** Expanding-window CV has only 3 folds (vs 5 in stratified) and each fold has different attack type distributions. This makes CV scores noisier — but correctly reflects the temporal reality of the problem. A model that performs well under these constraints is genuinely robust.

**Gap in literature:** None of the four academic sources address temporal splits with clustered attack types. Kamol et al. (2024) used random splits. Ness et al. (2025) used random CV on KDDCup'99. Our temporal approach is harder but honest.

## 3. Class Imbalance Handling

**Decision:** Three-layer approach: `scale_pos_weight` + confidence-based sample weights + no SMOTE. Inherited from 2.2 decisions with concrete implementation details.

### 3.1 Scale Positive Weight

- **LR/RF:** `class_weight='balanced'` (scikit-learn computes `n_samples / (n_classes × n_samples_per_class)` automatically)
- **XGBoost:** `scale_pos_weight` parameter — set to ratio `n_negatives / n_positives` (~83:1 in dataset)
- **LightGBM:** `is_unbalance=True` (auto-computes) or explicit `scale_pos_weight`

Optuna will tune `scale_pos_weight` in range [50, 150] for XGBoost/LightGBM rather than fixing at the exact ratio. The optimal weight depends on the interaction with other hyperparameters (depth, regularization).

### 3.2 Confidence-Based Sample Weights

Computed inline from `incident_labels.csv` confidence field: `high=1.0, medium=0.6, low=0.3`.

These are passed as `sample_weight` to the `.fit()` method. They multiply with the class weight, so a high-confidence malicious sample gets `83 × 1.0 = 83` effective weight, while a low-confidence one gets `83 × 0.3 = 24.9`.

### 3.3 No SMOTE

Rationale from 2.2 — only 592 malicious samples across 5 heterogeneous attack types. Interpolating between DDoS and credential stuffing produces synthetic examples that represent no real attack pattern.

**Trade-off:** We accept lower recall on rare attack types (zero_day_exploit has the fewest samples) in exchange for not contaminating the minority class.

## 4. Primary Metric: PR-AUC

**Decision:** Optimize and report PR-AUC as the primary metric. F1 and FPR as secondary.

**Why not accuracy?**
- Dataset is 98.8% negative. A model that predicts "benign" for everything gets 98.8% accuracy.

**Why not ROC-AUC?**
- ROC-AUC includes TNR (true negative rate) which is always near 1.0 with extreme imbalance. This inflates the score and masks poor recall.
- PR-AUC focuses entirely on the minority class: how precise is the model when it says "malicious" (precision) and how many actual attacks does it catch (recall)?

**Why not F1 alone?**
- F1 is threshold-dependent — a single number at a single threshold. PR-AUC summarizes performance across all thresholds.
- We report F1 as secondary because the assessment asks for it, and it's easy to interpret.

**FPR:** Reported because the assessment explicitly asks for it and it connects to the $2.50 false positive cost in section 3.3.

## 5. Hyperparameter Tuning (Optuna)

**Decision:** Bayesian optimization with Optuna, 10-30 trials per model, using expanding-window temporal CV PR-AUC as objective.

**Why Optuna over grid search?**
- Bayesian search (TPE sampler) finds better hyperparameters with fewer trials than grid search.
- MedianPruner early-stops unpromising trials based on intermediate CV fold results — saves compute.
- With 50k rows, individual trials are fast (seconds), so 80 trials takes minutes, not hours.

**Why Optuna over random search?**
- Random search is a strong baseline but TPE converges faster, especially for correlated hyperparameters (e.g., learning_rate and n_estimators).

**Why not just sensible defaults?**
- Default hyperparameters leave significant performance on the table, especially for the class imbalance parameters (scale_pos_weight) and regularization.
- Ness et al. (2025) used default parameters and achieved 0.85 test accuracy. With Optuna, we should exceed that.

### Search Spaces

**Logistic Regression** (10 trials — fast, small space):
- `C`: [1e-4, 1e2] log scale — regularization strength
- `l1_ratio`: [0.0, 1.0] — elasticnet mixing (0=L2, 1=L1). Uses `penalty='elasticnet'` with `solver='saga'`. Note: sklearn 1.8+ deprecates explicit `penalty` selection; a future iteration should migrate to using `l1_ratio` and `C` directly without the `penalty` parameter.

**Random Forest** (20 trials):
- `n_estimators`: [100, 500]
- `max_depth`: [5, 20]
- `min_samples_leaf`: [5, 50] — prevents overfitting to small minority leaf nodes
- `max_features`: sqrt, log2 — feature subsampling per split

**XGBoost** (30 trials):
- `n_estimators`: [100, 800]
- `max_depth`: [3, 10]
- `learning_rate`: [0.01, 0.3] log scale
- `subsample`: [0.6, 1.0] — row subsampling
- `colsample_bytree`: [0.5, 1.0] — feature subsampling
- `min_child_weight`: [1, 20] — minimum samples per leaf
- `scale_pos_weight`: [50, 150] — class imbalance
- `reg_alpha`: [1e-8, 10] log — L1 regularization
- `reg_lambda`: [1e-8, 10] log — L2 regularization

**LightGBM** (30 trials):
- `n_estimators`: [100, 800]
- `num_leaves`: [15, 127] — leaf-wise specific
- `max_depth`: [3, 10]
- `learning_rate`: [0.01, 0.3] log scale
- `subsample`: [0.6, 1.0]
- `colsample_bytree`: [0.5, 1.0]
- `min_child_samples`: [5, 50]
- `scale_pos_weight`: [50, 150]
- `reg_alpha`: [1e-8, 10] log
- `reg_lambda`: [1e-8, 10] log

**Optuna impact on edge deployment:** Zero. Optuna runs offline during training only. The final model artifact is identical regardless of how hyperparameters were found.

## 6. Feature Importance & Post-Training Pruning

**Decision:** Extract feature importances from all models, then prune the best model and retrain.

### 6.1 Importance Extraction Methods

| Model | Method | Why |
|---|---|---|
| Logistic Regression | Absolute coefficient values | Coefficients directly measure feature contribution to log-odds. |
| Random Forest | Gini importance (`feature_importances_`) | Default scikit-learn impurity-based importance. Fast and sufficient for the 36-feature set. |
| XGBoost | `feature_importances_` (gain-based) | Gain measures total loss reduction from splits on each feature. More meaningful than split count, which favors high-cardinality features. |
| LightGBM | `feature_importances_` (gain-based) | Same reasoning as XGBoost. |

**Actual top features (LGBM gain importance):** `ua_length`, `ua_entropy`, `hour_of_day`, `header_count`, `path_entropy`, `ip_1m_inter_request_time_mean`, `ip_30m_inter_request_time_mean`, `ip_1m_request_count`, `path_length`, `ip_5m_request_count`. Behavioral features (user-agent characteristics, timing, volume) now dominate, not identity features. This is a direct consequence of removing TLS-granularity features that acted as label proxies.

### 6.2 Pruning Process

1. Train best model with all 36 features (15 per-request + 21 IP-level session) on the training set.
2. Rank features by gain importance.
3. Compute cumulative importance at multiple thresholds (90%, 95%, 99%) to compare aggressiveness.
4. Drop features below the chosen cutoff.
5. Retrain with pruned feature set using same hyperparameters.
6. Compare PR-AUC and per-attack recall on the held-out test set.

**Pruning results (LGBM):**
- Pruning from 36 to 30 features: PR-AUC drops by only 0.024 (0.8138 to 0.7894)

**Analysis:** Pruning is now nearly lossless. Without TLS features, the model distributes importance more evenly across behavioral features. Removing the bottom ~17% of features has minimal impact because the remaining features capture the core behavioral signals (user-agent, timing, volume). The model no longer depends on a fragile set of TLS identity features.

### 6.3 Per-Attack-Type Breakdown

Per-attack recall on the test set (test contains zero_day_exploit, credential_stuffing, ddos_l7):

| Model | zero_day_exploit | credential_stuffing | ddos_l7 |
|---|---|---|---|
| LR | 0.0 | 0.0087 | 0.0 |
| RF | 0.0 | 0.0 | 0.0 |
| XGB | 0.0 | 0.8551 | 0.0 |
| LGBM | 0.0 | 0.8696 | 0.0 |

Credential_stuffing is the only well-detected attack type (0.87 recall with LGBM) because it has a distinctive behavioral pattern (repeated login attempts with consistent timing). DDoS L7 has 0% recall because the attack is distributed across many IPs — no single IP accumulates enough session-level signal for detection. Zero_day_exploit has 0% recall because the model has never seen this attack type in training and has only 6 test samples.

After pruning (LGBM, 36 to 30 features): credential_stuffing recall stays at 0.87, ddos_l7 stays at 0.0, zero_day_exploit stays at 0.0. The pruned model preserves detection capability — it no longer depends on a fragile feature set.

### 6.4 Connection to 2.2 Validation

Compare actual feature importance rankings with the predicted rankings from section 10 of `2.2-feature-engineering-decisions.md`. The baseline uses 36 features (15 per-request + 21 IP-level session with causal windowing). With TLS features removed, behavioral features dominate: user-agent characteristics (`ua_length`, `ua_entropy`), temporal patterns (`hour_of_day`, `ip_*_inter_request_time_mean`), and request structure (`header_count`, `path_entropy`). This validates the feature design — genuine behavioral signals drive detection.

### 6.5 Connection to 2.4

The pruned model is what gets exported for edge deployment. Fewer features = fewer tree splits = smaller WASM binary = faster inference. The pruning analysis in 2.3 directly feeds into the deployment feasibility analysis in 2.4.

## 7. Threshold Analysis

**Decision:** Report metrics at default threshold (0.5) AND find cost-optimal threshold using the assessment's cost structure.

### 7.1 Default Threshold (0.5)

Standard comparison table for all 4 models (temporal split, 36 features: 15 per-request + 21 IP-level session with causal windowing):

| Model | CV PR-AUC | Test PR-AUC | Precision | Recall | F1 | FPR |
|---|---|---|---|---|---|---|
| Logistic Regression | 0.5432 | 0.0123 | 0.2857 | 0.0082 | 0.0159 | 0.000239 |
| Random Forest | 0.7771 | 0.8086 | 0.0000 | 0.0000 | 0.0000 | 0.000000 |
| XGBoost | 0.8894 | 0.6206 | 1.0000 | 0.5224 | 0.6863 | 0.000000 |
| LightGBM | 0.9441 | 0.8138 | 1.0000 | 0.4898 | 0.6575 | 0.000000 |

> **Context:** With TLS features removed (see 2.2 section 11 for rationale), the model relies on genuine behavioral features. The PR-AUC of 0.81 for LGBM represents honest behavioral detection capability — a realistic baseline for production deployment.

### 7.2 Cost-Optimal Threshold

Using the cost structure from section 3.3 of the assessment:
- FP cost (blocking legitimate request): $2.50
- FN cost (attack reaching origin): $0.10

**Expected cost per request at threshold t:**

```
Cost(t) = FPR(t) × (1 - π) × $2.50  +  FNR(t) × π × $0.10
```

Where π = attack base rate.

- Dataset base rate: ~1.2% (592/50000). Production base rate: 0.05%.
- Compute cost curve across thresholds [0.01, 0.99] for best model (LGBM).
- Find t* minimizing total expected cost.

**Results (LGBM):**

| Scenario | Threshold | Precision | Recall | F1 | FPR |
|---|---|---|---|---|---|
| Default | 0.500 | 1.0000 | 0.4898 | 0.6575 | 0.000000 |
| Normal costs ($2.50/$0.10) | 0.144 | 1.0000 | 0.5102 | 0.6757 | 0.000000 |
| High FN costs ($2.50/$5.00) | 0.010 | 0.9700 | 0.5510 | 0.7027 | 0.000000 |

The probability distribution is **no longer bimodal** — unlike the previous model with TLS features, the model now has a genuine precision-recall tradeoff. Lowering the threshold from 0.5 to 0.144 gains only marginal recall (0.49 to 0.51) because the model's probability scores for many malicious samples cluster in a narrow range. Lowering further to 0.01 pushes recall to 0.55 with precision at 0.97 — still high precision, but the gains in recall are modest. This reflects the genuine difficulty of behavioral detection without identity shortcuts.

**Why do this in 2.3?**
- The assessment asks for precision/recall/F1/FPR — these depend on threshold choice.
- Reporting only at 0.5 is naive for extreme imbalance.
- Shows awareness of the precision-recall trade-off. Section 3.3 then deepens the math.
- Threshold optimization is genuinely important when the model has a real tradeoff (unlike the previous bimodal distribution with TLS features).

## 9. Expected vs Actual Outcomes

Based on 36 features (15 per-request + 21 IP-level session with causal windowing):

- **LGBM is the best model with PR-AUC 0.81** — LGBM achieves the highest test PR-AUC (0.8138) and the best F1 (0.6575 at default threshold). This is a genuine behavioral detection score, not inflated by TLS identity shortcuts. The model correctly learns user-agent characteristics, timing patterns, and request structure.

- **Credential_stuffing is well detected (0.87 recall)** — The behavioral pattern of credential stuffing (repeated login attempts, consistent timing, bot-like user-agents) is learnable from IP-level session features. This is the model's strongest detection capability.

- **ddos_l7 is NOT detected (0% recall)** — DDoS L7 in this dataset is distributed across many source IPs. No single IP accumulates enough session-level signal (request volume, timing regularity) for detection. This is a fundamental limitation of per-IP session features for distributed attacks. Detection requires architectural solutions: cross-node rate limiting, global IP aggregation, or dedicated DDoS mitigation infrastructure.

- **zero_day_exploit is NOT detected (0% recall)** — Only 6 samples exist, all in the test set (never seen in training). Without TLS fingerprint shortcuts, the model has no basis for detecting a truly novel attack type. This validates the assessment's question 3.2: unsupervised/hybrid approaches are necessary for zero-day detection.

- **LR performs terribly (PR-AUC 0.01)** — Logistic regression cannot capture the non-linear behavioral patterns that distinguish attacks from benign traffic. The interaction effects between features (e.g., high request count + low timing variance + bot-like UA = attack) are exactly what tree splits exploit and linear models miss.

- **RF has good PR-AUC (0.81) but 0% recall at threshold 0.5** — RF achieves PR-AUC comparable to LGBM but has poor probability calibration. All true positive probabilities fall below the 0.5 decision boundary, resulting in zero predictions of "malicious" at the default threshold. This makes RF unsuitable for deployment without careful threshold tuning.

- **Feature pruning is nearly lossless** — Pruning from 36 to 30 features causes only -0.024 PR-AUC (0.8138 to 0.7894). Without TLS features, the model distributes importance across genuine behavioral features that each contribute independent signal. The pruned model is a viable deployment candidate.

- **The model has a real precision-recall tradeoff** — The probability distribution is no longer bimodal. Default threshold 0.5 gives precision 1.0 but only 0.49 recall. Lowering to 0.144 gives 0.51 recall; lowering to 0.01 gives 0.55 recall with 0.97 precision. This reflects the genuine difficulty of behavioral detection.

- **Best LGBM hyperparameters** — n_estimators=589, num_leaves=66, max_depth=8, learning_rate=0.0380, subsample=0.6867, colsample_bytree=0.6411, min_child_samples=44, scale_pos_weight=138.27. The high scale_pos_weight (138x) reflects the extreme class imbalance and the model's need to heavily penalize missed attacks.

## 10. Data Leakage Lessons Learned

During development, five sources of data leakage or circularity were discovered. Two were fixed; three were addressed by exclusion or removal:

1. **Server-response per-request features (3 features, excluded):** `status_code_group`, `response_time_ms`, and `body_size_bytes` are server-side response attributes. At inference time, the model classifies the request *before* the server responds. Including them produced PR-AUC 1.0 because status codes (403/429) directly encode the server's own decisions. These are listed in `META_COLUMNS` in `src/model.py`.

2. **Session feature lookahead (fixed):** The original `compute_session_features()` used `groupby().transform()` over the full dataset, including future requests and test-set rows. **Resolved** by reimplementing with causal windowing: `cumcount()` for request counts, `expanding().mean().shift(1)` for running averages, and custom `_causal_nunique`/`_causal_entropy` functions for diversity metrics. Each request now sees only prior requests within its time window. Verified by unit tests.

3. **Frequency encodings (removed):** `compute_frequency_encoding()` and the 4 `_freq` columns were fully removed from the codebase. They computed value frequencies over the full dataset, leaking test distribution into training, and were already excluded from model features.

4. **Server-response session features (18 features, excluded):** Aggregates of past `status_code`, `response_time_ms`, and `body_size_bytes` per source (IP-level only). Even with causal windowing, these encode past WAF decisions, creating a circular dependency with labels. Excluded via `RESPONSE_SESSION_SUFFIXES` in `src/model.py`.

5. **TLS-granularity session features (39 features, removed):** Each TLS fingerprint in the synthetic dataset is 100% malicious or 100% benign, and labels are partially assigned by TLS fingerprint. This creates a circular dependency — the model learns the label lookup table, not behavior. All TLS-granularity session features were removed entirely. With TLS features, LGBM achieved PR-AUC 1.0; without them, PR-AUC drops to 0.81, representing the model's genuine behavioral detection capability. See 2.2 section 11 for the full analysis.

The baseline model uses 36 features: 15 per-request + 21 IP-level session features with causal windowing. Server-response features (3 per-request + 18 session) and TLS-granularity features (39) are excluded. Frequency-encoded categoricals were fully removed from the codebase.

### Key Insight: Honest Evaluation Over Inflated Metrics

The most important design decision was removing TLS-granularity session features despite their producing perfect PR-AUC. A model that achieves PR-AUC 1.0 by memorizing a label proxy provides zero value in production — it would fail the moment attackers randomize their TLS fingerprints. The 0.81 PR-AUC with behavioral features only is an honest measure of what the model can do, and it reveals genuine gaps (distributed attacks, novel attack types) that inform the production architecture.
