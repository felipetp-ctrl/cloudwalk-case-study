## 1. Model Selection

**Decision:** Train and compare four models in a simple â†’ complex progression: Logistic Regression â†’ Random Forest â†’ XGBoost â†’ LightGBM.

**Why four models?**
- LR establishes a linear lower bound. If trees barely beat LR, the non-linear signal is weak and the simpler model wins for edge deployment.
- RF is a bagging baseline â€” shows whether boosting's sequential error correction actually matters or if parallel trees suffice.
- XGBoost and LightGBM are the real contenders. Comparing both lets us pick the one with the best accuracy-to-size ratio for WASM export.

**Why not deep learning?**
- WASM runtime, no GPU, <5ms latency budget. LSTMs and transformers are out.
- Dawadi et al. (2023) achieved 97.57% DDoS detection with LSTM but reports no latency benchmarks â€” inference requires significant compute.
- Tree models achieve comparable performance on tabular data with microsecond-level inference (Mayhew et al., 2015 reports SVM classification "in the microsecond range").

**Why not SVM?**
- SVMs scale poorly with dataset size (O(nÂ˛) to O(nÂł) training complexity). With 50k samples and Optuna tuning, training time becomes prohibitive.
- LightGBM/XGBoost match or exceed SVM accuracy on tabular data with faster training.
- SVM lacks native feature importance â€” harder to do the pruning analysis.

**Why not Isolation Forest?**
- Ness et al. (2025) found Isolation Forest achieved only 0.4 test accuracy on KDDCup'99, far below supervised methods.
- We have labels â€” using unsupervised methods wastes information.

**Academic support:**
- Ness et al. (2025, IEEE Access): Compared Isolation Forest, Naive Bayes, XGBoost, LightGBM, SVM on KDDCup'99. LightGBM achieved training accuracy 1.0 and test accuracy 0.85. XGBoost: 0.99/0.83.
- Kamol et al. (2024, ICCIT): Used DBSCAN clustering on web access logs for credential stuffing â€” achieved 90.56% accuracy. Validates IP-level aggregation but unsupervised only.
- Dawadi et al. (2023, Sensors): LSTM-based WAF with layered architecture. 97.57% DDoS accuracy. Too heavy for our constraints â€” serves as contrast justification for trees.
- Mayhew et al. (2015, MILCOM): BBAC system using SVM + clustering for HTTP traffic behavioral analysis. Pipeline mirrors ours: feature extraction â†’ enrichment â†’ classification. Real-time latency ~10ms.

## 2. Split Strategy

**Decision:** Temporal train/test split (train days 6-9, test days 10-12) + expanding-window temporal CV for Optuna tuning. No future data leaks into training.

### 2.1 Temporal Split Design

| Day | Attack Types | Split |
|---|---|---|
| Jan 6 | None (0 malicious) | Train |
| Jan 7 | credential_stuffing (111), scanner (25) | Train |
| Jan 8 | api_abuse (34) | Train |
| Jan 9 | api_abuse (31), ddos_l7 (146) | Train |
| Jan 10 | zero_day_exploit (6) | Test |
| Jan 11 | credential_stuffing (138) | Test |
| Jan 12 | ddos_l7 (101) | Test |

**Cutoff: Jan 10.** Train sees 4 attack types (credential_stuffing, scanner, api_abuse, ddos_l7). Test sees 3 types (zero_day_exploit, credential_stuffing, ddos_l7). This is realistic: in production, the model encounters future traffic that may include novel attack types never seen during training.

**Key properties:**
- `zero_day_exploit` appears only in test â€” the model has never seen this attack type. With TLS features, the model detects it via its unique fingerprint (`ja3_c0ffee000003`), but this is tool identification, not behavioral generalization. The unsupervised/hybrid strategy discussed in assessment 3.2 remains the robust approach for truly novel attacks.
- `scanner` and `api_abuse` appear only in train â€” tests whether the model overfits to specific attack type signals.
- `credential_stuffing` and `ddos_l7` appear in both â€” allows evaluation of generalization across time for known attack types.

### 2.2 Split Layout

| Usage | Method | Purpose |
|---|---|---|
| Train/Test | Temporal cutoff at Jan 10 (~58/42) | Final evaluation |
| CV Folds 1-3 | Expanding window (6â†’7, 6-7â†’8, 6-8â†’9) | Optuna tuning |

**Trade-off:** Expanding-window CV has only 3 folds (vs 5 in stratified) and each fold has different attack type distributions. This makes CV scores noisier â€” but correctly reflects the temporal reality of the problem. A model that performs well under these constraints is genuinely robust.

**Gap in literature:** None of the four academic sources address temporal splits with clustered attack types. Kamol et al. (2024) used random splits. Ness et al. (2025) used random CV on KDDCup'99. Our temporal approach is harder but honest.

## 3. Class Imbalance Handling

**Decision:** Three-layer approach: `scale_pos_weight` + confidence-based sample weights + no SMOTE. Inherited from 2.2 decisions with concrete implementation details.

### 3.1 Scale Positive Weight

- **LR/RF:** `class_weight='balanced'` (scikit-learn computes `n_samples / (n_classes Ă— n_samples_per_class)` automatically)
- **XGBoost:** `scale_pos_weight` parameter â€” set to ratio `n_negatives / n_positives` (~83:1 in dataset)
- **LightGBM:** `is_unbalance=True` (auto-computes) or explicit `scale_pos_weight`

Optuna will tune `scale_pos_weight` in range [50, 150] for XGBoost/LightGBM rather than fixing at the exact ratio. The optimal weight depends on the interaction with other hyperparameters (depth, regularization).

### 3.2 Confidence-Based Sample Weights

Computed inline from `incident_labels.csv` confidence field: `high=1.0, medium=0.6, low=0.3`.

These are passed as `sample_weight` to the `.fit()` method. They multiply with the class weight, so a high-confidence malicious sample gets `83 Ă— 1.0 = 83` effective weight, while a low-confidence one gets `83 Ă— 0.3 = 24.9`.

### 3.3 No SMOTE

Rationale from 2.2 â€” only 592 malicious samples across 5 heterogeneous attack types. Interpolating between DDoS and credential stuffing produces synthetic examples that represent no real attack pattern.

**Trade-off:** We accept lower recall on rare attack types (zero_day_exploit has the fewest samples) in exchange for not contaminating the minority class.

## 4. Primary Metric: PR-AUC

**Decision:** Optimize and report PR-AUC as the primary metric. F1 and FPR as secondary.

**Why not accuracy?**
- Dataset is 98.8% negative. A model that predicts "benign" for everything gets 98.8% accuracy.

**Why not ROC-AUC?**
- ROC-AUC includes TNR (true negative rate) which is always near 1.0 with extreme imbalance. This inflates the score and masks poor recall.
- PR-AUC focuses entirely on the minority class: how precise is the model when it says "malicious" (precision) and how many actual attacks does it catch (recall)?

**Why not F1 alone?**
- F1 is threshold-dependent â€” a single number at a single threshold. PR-AUC summarizes performance across all thresholds.
- We report F1 as secondary because the assessment asks for it, and it's easy to interpret.

**FPR:** Reported because the assessment explicitly asks for it and it connects to the $2.50 false positive cost in section 3.3.

## 5. Hyperparameter Tuning (Optuna)

**Decision:** Bayesian optimization with Optuna, 10-30 trials per model, using expanding-window temporal CV PR-AUC as objective.

**Why Optuna over grid search?**
- Bayesian search (TPE sampler) finds better hyperparameters with fewer trials than grid search.
- MedianPruner early-stops unpromising trials based on intermediate CV fold results â€” saves compute.
- With 50k rows, individual trials are fast (seconds), so 80 trials takes minutes, not hours.

**Why Optuna over random search?**
- Random search is a strong baseline but TPE converges faster, especially for correlated hyperparameters (e.g., learning_rate and n_estimators).

**Why not just sensible defaults?**
- Default hyperparameters leave significant performance on the table, especially for the class imbalance parameters (scale_pos_weight) and regularization.
- Ness et al. (2025) used default parameters and achieved 0.85 test accuracy. With Optuna, we should exceed that.

### Search Spaces

**Logistic Regression** (10 trials â€” fast, small space):
- `C`: [1e-4, 1e2] log scale â€” regularization strength
- `l1_ratio`: [0.0, 1.0] â€” elasticnet mixing (0=L2, 1=L1). Uses `penalty='elasticnet'` with `solver='saga'`. Note: sklearn 1.8+ deprecates explicit `penalty` selection; a future iteration should migrate to using `l1_ratio` and `C` directly without the `penalty` parameter.

**Random Forest** (20 trials):
- `n_estimators`: [100, 500]
- `max_depth`: [5, 20]
- `min_samples_leaf`: [5, 50] â€” prevents overfitting to small minority leaf nodes
- `max_features`: sqrt, log2 â€” feature subsampling per split

**XGBoost** (30 trials):
- `n_estimators`: [100, 800]
- `max_depth`: [3, 10]
- `learning_rate`: [0.01, 0.3] log scale
- `subsample`: [0.6, 1.0] â€” row subsampling
- `colsample_bytree`: [0.5, 1.0] â€” feature subsampling
- `min_child_weight`: [1, 20] â€” minimum samples per leaf
- `scale_pos_weight`: [50, 150] â€” class imbalance
- `reg_alpha`: [1e-8, 10] log â€” L1 regularization
- `reg_lambda`: [1e-8, 10] log â€” L2 regularization

**LightGBM** (30 trials):
- `n_estimators`: [100, 800]
- `num_leaves`: [15, 127] â€” leaf-wise specific
- `max_depth`: [3, 10]
- `learning_rate`: [0.01, 0.3] log scale
- `subsample`: [0.6, 1.0]
- `colsample_bytree`: [0.5, 1.0]
- `min_child_samples`: [5, 50]
- `scale_pos_weight`: [50, 150]
- `reg_alpha`: [1e-8, 10] log
- `reg_lambda`: [1e-8, 10] log

**Optuna impact on edge deployment:** Zero. Optuna runs offline during training only. The final model artifact is identical regardless of how hyperparameters were found.

## 6. Feature Importance & Post-Training Pruning

**Decision:** Extract feature importances from all models, then prune the best model and retrain.

### 6.1 Importance Extraction Methods

| Model | Method | Why |
|---|---|---|
| Logistic Regression | Absolute coefficient values | Coefficients directly measure feature contribution to log-odds. |
| Random Forest | Gini importance (`feature_importances_`) | Default scikit-learn impurity-based importance. Fast and sufficient for the 57-feature set. |
| XGBoost | `feature_importances_` (gain-based) | Gain measures total loss reduction from splits on each feature. More meaningful than split count, which favors high-cardinality features. |
| LightGBM | `feature_importances_` (gain-based) | Same reasoning as XGBoost. |

### 6.2 Pruning Process

1. Train best model with all 57 features (15 per-request + 42 session) on the training set.
2. Rank features by gain importance.
3. Compute cumulative importance at multiple thresholds (90%, 95%, 99%) to compare aggressiveness.
4. Drop features below the chosen cutoff.
5. Retrain with pruned feature set using same hyperparameters.
6. Compare PR-AUC and per-attack recall on the held-out test set.

**Pruning results (LGBM):**
- 90% threshold: 20/57 features kept â†’ PR-AUC drops significantly (model loses critical TLS features)
- 95% threshold: 26/57 features kept â†’ PR-AUC partially recovers
- 99% threshold: 32/57 features kept â†’ PR-AUC 0.879 (-0.121 from full model)

**Analysis:** Unlike the prior 15-feature baseline where pruning was nearly lossless, pruning with 57 features causes notable degradation. This is because the model distributes importance across many TLS-granularity features that each capture a different aspect of the same underlying signal (TLS fingerprint identity). Removing any subset breaks the redundancy that allows perfect classification. The 25 dropped features (mostly IP-granularity session features) contribute <1% of total gain â€” confirming that IP-based behavioral features are weak discriminators in this dataset.

### 6.3 Per-Attack-Type Breakdown

Per-attack recall on the test set (test contains zero_day_exploit, credential_stuffing, ddos_l7):

| Model | zero_day_exploit | credential_stuffing | ddos_l7 |
|---|---|---|---|
| LR | 1.0 | 1.0 | 0.822 |
| RF | 0.333 | 0.978 | 0.0 |
| XGB | 0.333 | 1.0 | 1.0 |
| LGBM | 1.0 | 1.0 | 1.0 |

RF's ddos_l7 recall of 0.0 despite PR-AUC 1.0 reflects poor probability calibration at threshold 0.5 â€” the true positive probabilities are just below the decision boundary. LGBM achieves perfect recall across all attack types.

After pruning (LGBM, 99% threshold, 32/57 features): credential_stuffing recall collapses to 0.0, ddos_l7 to 0.03, while zero_day_exploit remains 1.0. This confirms that credential_stuffing and ddos_l7 detection depends on specific TLS features that get pruned.

### 6.4 Connection to 2.2 Validation

Compare actual feature importance rankings with the predicted rankings from section 10 of `2.2-feature-engineering-decisions.md`. The baseline uses 57 features (15 per-request + 42 request-side session with causal windowing). The tree model's feature importance reveals that TLS-granularity session features dominate (14 of top 20), which led to the TLS fingerprint analysis documented in 2.2 section 11.

### 6.5 Connection to 2.4

The pruned model is what gets exported for edge deployment. Fewer features = fewer tree splits = smaller WASM binary = faster inference. The pruning analysis in 2.3 directly feeds into the deployment feasibility analysis in 2.4.

## 7. Threshold Analysis

**Decision:** Report metrics at default threshold (0.5) AND find cost-optimal threshold using the assessment's cost structure.

### 7.1 Default Threshold (0.5)

Standard comparison table for all 4 models (temporal split, 57 features: 15 per-request + 42 session with causal windowing):

| Model | CV PR-AUC | PR-AUC | ROC-AUC | Precision | Recall | F1 | FPR |
|---|---|---|---|---|---|---|---|
| Logistic Regression | 0.9924 | 0.9589 | 0.9865 | 1.0000 | 0.9265 | 0.9619 | 0.0000 |
| Random Forest | 0.9280 | 1.0000 | 1.0000 | 1.0000 | 0.5592 | 0.7173 | 0.0000 |
| XGBoost | 0.7793 | 1.0000 | 1.0000 | 1.0000 | 0.9837 | 0.9918 | 0.0000 |
| LightGBM | 0.8971 | 1.0000 | 1.0000 | 1.0000 | 1.0000 | 1.0000 | 0.0000 |

> **Important context:** The perfect PR-AUC scores on tree models are driven by TLS fingerprints being perfect label proxies in this synthetic dataset â€” see 2.2 section 11 for the full analysis. The model correctly finds the strongest signal (TLS identity), but this signal would degrade in production when attackers spoof TLS fingerprints.

### 7.2 Cost-Optimal Threshold

Using the cost structure from section 3.3 of the assessment:
- FP cost (blocking legitimate request): $2.50
- FN cost (attack reaching origin): $0.10

**Expected cost per request at threshold t:**

```
Cost(t) = FPR(t) Ă— (1 - Ď€) Ă— $2.50  +  FNR(t) Ă— Ď€ Ă— $0.10
```

Where Ď€ = attack base rate.

- Dataset base rate: ~1.2% (592/50000). Production base rate: 0.05%.
- Compute cost curve across thresholds [0.01, 0.99] for best model (LGBM).
- Find t* minimizing total expected cost.

**Results (LGBM):**

| Scenario | Threshold | Precision | Recall | F1 | FPR |
|---|---|---|---|---|---|
| Default | 0.500 | 1.0000 | 1.0000 | 1.0000 | 0.000000 |
| Normal costs ($2.50/$0.10) | 0.022 | 1.0000 | 1.0000 | 1.0000 | 0.000000 |
| High FN costs ($2.50/$5.00) | 0.022 | 1.0000 | 1.0000 | 1.0000 | 0.000000 |

All cost scenarios converge to perfect performance because the model's probability distribution is extremely bimodal: benign predictions cluster near 0 (mean=0.0001, max=0.021) and malicious predictions cluster near 1 (mean=0.949, min=0.762). Any threshold between 0.02 and 0.76 achieves perfect classification. This extreme separation is a consequence of TLS fingerprints being perfect label proxies (see 2.2 section 11).

**Why do this in 2.3?**
- The assessment asks for precision/recall/F1/FPR â€” these depend on threshold choice.
- Reporting only at 0.5 is naive for extreme imbalance.
- Shows awareness of the precision-recall trade-off. Section 3.3 then deepens the math.
- In production, where TLS signal degrades, threshold optimization becomes genuinely important.

## 9. Expected vs Actual Outcomes

Based on 57 features (15 per-request + 42 session with causal windowing):

- **Tree models achieve perfect PR-AUC** â€” All three tree models (RF, XGB, LGBM) reach PR-AUC 1.0 on the test set. Investigation revealed this is driven by TLS fingerprints being perfect label proxies in the synthetic dataset (see 2.2 section 11), not by genuinely learning behavioral patterns. The model correctly exploits the strongest signal available.

- **LR achieves near-perfect test performance but lower train performance** â€” LR test PR-AUC (0.9589) exceeds train PR-AUC (0.3562). This test > train anomaly is unusual but explainable: test benign traffic has wider separation from malicious on TLS session features (`tls_30m_request_count` benign mean = 14,128 in test vs 5,194 in train; malicious remains ~180 in both). LR, as a linear model, benefits from this wider margin.

- **LGBM is the best model** â€” Selected by (PR-AUC, F1) tiebreaker: PR-AUC 1.0 ties with RF and XGB, but LGBM achieves F1 1.0 (RF F1 0.7173, XGB F1 0.9918). LGBM detects all three test attack types with 100% recall.

- **zero_day_exploit is detectable via TLS fingerprint** â€” Unlike the prior 15-feature baseline where zero_day_exploit had 0% recall (no training samples for this type), the model now detects it via the TLS session features. The 6 zero_day_exploit requests share a unique TLS fingerprint (`ja3_c0ffee000003`) that produces distinctive session feature values (very small TLS group). LGBM and LR detect all 6; RF and XGB detect 2/6. This is **not** generalization to unseen attack types â€” the model identifies the tool, not the behavior.

- **ddos_l7 detection improved dramatically** â€” From 0.13 recall (15-feature baseline) to 1.0 recall (LGBM). Session features enabled volumetric pattern detection. However, most of this signal comes from TLS-granularity features rather than IP-granularity timing/volume features.

- **RF has worst recall despite perfect PR-AUC** â€” RF achieves PR-AUC 1.0 but only 0.56 recall at threshold 0.5 (ddos_l7 recall = 0.0). This is because RF's probability estimates are less calibrated â€” many true positives get probability scores just below 0.5.

- **Feature pruning is lossy** â€” Unlike the prior 15-feature baseline where 99% pruning was nearly lossless (-0.0006 PR-AUC), pruning 57 â†’ 32 features causes PR-AUC 0.879 (-0.121). The model distributes importance across many TLS features; removing any subset breaks the redundancy needed for perfect classification. The 25 dropped features are mostly IP-granularity session features, confirming IP-based behavioral signal is weak in this dataset.

- **Cost-optimal threshold is very low** â€” Both cost scenarios converge to threshold 0.022, achieving perfect metrics. The model's probability distribution is extremely bimodal (benign mean=0.0001, malicious min=0.762), so any threshold in [0.02, 0.76] works. In production with degraded TLS signal, threshold optimization would become genuinely important.

## 10. Data Leakage Lessons Learned

During development, four sources of data leakage were discovered. Two were fixed; two were addressed by exclusion:

1. **Server-response per-request features (3 features, excluded):** `status_code_group`, `response_time_ms`, and `body_size_bytes` are server-side response attributes. At inference time, the model classifies the request *before* the server responds. Including them produced PR-AUC 1.0 because status codes (403/429) directly encode the server's own decisions. These are listed in `META_COLUMNS` in `src/model.py`.

2. **Session feature lookahead (fixed):** The original `compute_session_features()` used `groupby().transform()` over the full dataset, including future requests and test-set rows. **Resolved** by reimplementing with causal windowing: `cumcount()` for request counts, `expanding().mean().shift(1)` for running averages, and custom `_causal_nunique`/`_causal_entropy` functions for diversity metrics. Each request now sees only prior requests within its time window. Verified by unit tests.

3. **Frequency encodings (4 features, excluded):** `compute_frequency_encoding()` computes value frequencies over the full dataset, leaking test distribution into training. Remains excluded; proper fix requires train-only fitting via sklearn pipelines.

4. **Server-response session features (36 features, excluded):** Aggregates of past `status_code`, `response_time_ms`, and `body_size_bytes` per source. Even with causal windowing, these encode past WAF decisions, creating a circular dependency with labels. Excluded via `RESPONSE_SESSION_SUFFIXES` in `src/model.py`.

The baseline model uses 57 features: 15 per-request + 42 request-side session features with causal windowing. The remaining 36 features (server-response session) are computed by the pipeline but excluded from the model.

### Key Insight: Synthetic Data Limitations

The most important finding is not a leakage bug but a dataset characteristic: TLS fingerprints are unique per attack type in the synthetic data, making TLS-granularity session features perfect label proxies. The model's PR-AUC 1.0 is technically correct (no leakage), but reflects **tool identification** rather than **behavioral detection**. In production with TLS fingerprint spoofing, performance would degrade to the IP + per-request feature floor (~0.71 PR-AUC). See 2.2 section 11 for the full analysis.
