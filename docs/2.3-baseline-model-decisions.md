# 2.3 Baseline Model — Decision Dump

This document captures every design choice, trade-off, and justification for the baseline model pipeline. It serves as the reasoning backbone for the implementation and writeup.

---

## 1. Model Selection

**Decision:** Train and compare four models in a simple → complex progression: Logistic Regression → Random Forest → XGBoost → LightGBM.

**Why four models?**
- LR establishes a linear lower bound. If trees barely beat LR, the non-linear signal is weak and the simpler model wins for edge deployment.
- RF is a bagging baseline — shows whether boosting's sequential error correction actually matters or if parallel trees suffice.
- XGBoost and LightGBM are the real contenders. Comparing both lets us pick the one with the best accuracy-to-size ratio for WASM export.

**Why not deep learning?**
- WASM runtime, no GPU, <5ms latency budget. LSTMs and transformers are out.
- Dawadi et al. (2023) achieved 97.57% DDoS detection with LSTM but reports no latency benchmarks — inference requires significant compute.
- Tree models achieve comparable performance on tabular data with microsecond-level inference (Mayhew et al., 2015 reports SVM classification "in the microsecond range").

**Why not SVM?**
- SVMs scale poorly with dataset size (O(n²) to O(n³) training complexity). With 50k samples and Optuna tuning, training time becomes prohibitive.
- LightGBM/XGBoost match or exceed SVM accuracy on tabular data with faster training.
- SVM lacks native feature importance — harder to do the pruning analysis.

**Why not Isolation Forest?**
- Ness et al. (2025) found Isolation Forest achieved only 0.4 test accuracy on KDDCup'99, far below supervised methods.
- We have labels — using unsupervised methods wastes information.

**Academic support:**
- Ness et al. (2025, IEEE Access): Compared Isolation Forest, Naive Bayes, XGBoost, LightGBM, SVM on KDDCup'99. LightGBM achieved training accuracy 1.0 and test accuracy 0.85. XGBoost: 0.99/0.83.
- Kamol et al. (2024, ICCIT): Used DBSCAN clustering on web access logs for credential stuffing — achieved 90.56% accuracy. Validates IP-level aggregation but unsupervised only.
- Dawadi et al. (2023, Sensors): LSTM-based WAF with layered architecture. 97.57% DDoS accuracy. Too heavy for our constraints — serves as contrast justification for trees.
- Mayhew et al. (2015, MILCOM): BBAC system using SVM + clustering for HTTP traffic behavioral analysis. Pipeline mirrors ours: feature extraction → enrichment → classification. Real-time latency ~10ms.

## 2. Split Strategy

**Decision:** Temporal train/test split (train days 6-9, test days 10-12) + expanding-window temporal CV for Optuna tuning. No future data leaks into training.

### 2.1 Temporal Split Design

| Day | Attack Types | Split |
|---|---|---|
| Jan 6 | None (0 malicious) | Train |
| Jan 7 | credential_stuffing (111), scanner (25) | Train |
| Jan 8 | api_abuse (34) | Train |
| Jan 9 | api_abuse (31), ddos_l7 (146) | Train |
| Jan 10 | zero_day_exploit (6) | Test |
| Jan 11 | credential_stuffing (138) | Test |
| Jan 12 | ddos_l7 (101) | Test |

**Cutoff: Jan 10.** Train sees 4 attack types (credential_stuffing, scanner, api_abuse, ddos_l7). Test sees 3 types (zero_day_exploit, credential_stuffing, ddos_l7). This is realistic: in production, the model encounters future traffic that may include novel attack types never seen during training.

**Key properties:**
- `zero_day_exploit` appears only in test — the model has never seen this attack type. This directly demonstrates the limitation that motivates the unsupervised/hybrid strategy discussed in assessment 3.2.
- `scanner` and `api_abuse` appear only in train — tests whether the model overfits to specific attack type signals.
- `credential_stuffing` and `ddos_l7` appear in both — allows evaluation of generalization across time for known attack types.

### 2.2 Split Layout

| Usage | Method | Purpose |
|---|---|---|
| Train/Test | Temporal cutoff at Jan 10 (~58/42) | Final evaluation |
| CV Folds 1-3 | Expanding window (6→7, 6-7→8, 6-8→9) | Optuna tuning |

**Trade-off:** Expanding-window CV has only 3 folds (vs 5 in stratified) and each fold has different attack type distributions. This makes CV scores noisier — but correctly reflects the temporal reality of the problem. A model that performs well under these constraints is genuinely robust.

**Gap in literature:** None of the four academic sources address temporal splits with clustered attack types. Kamol et al. (2024) used random splits. Ness et al. (2025) used random CV on KDDCup'99. Our temporal approach is harder but honest.

## 3. Class Imbalance Handling

**Decision:** Three-layer approach: `scale_pos_weight` + confidence-based sample weights + no SMOTE. Inherited from 2.2 decisions with concrete implementation details.

### 3.1 Scale Positive Weight

- **LR/RF:** `class_weight='balanced'` (scikit-learn computes `n_samples / (n_classes × n_samples_per_class)` automatically)
- **XGBoost:** `scale_pos_weight` parameter — set to ratio `n_negatives / n_positives` (~83:1 in dataset)
- **LightGBM:** `is_unbalance=True` (auto-computes) or explicit `scale_pos_weight`

Optuna will tune `scale_pos_weight` in range [50, 150] for XGBoost/LightGBM rather than fixing at the exact ratio. The optimal weight depends on the interaction with other hyperparameters (depth, regularization).

### 3.2 Confidence-Based Sample Weights

Computed inline from `incident_labels.csv` confidence field: `high=1.0, medium=0.6, low=0.3`.

These are passed as `sample_weight` to the `.fit()` method. They multiply with the class weight, so a high-confidence malicious sample gets `83 × 1.0 = 83` effective weight, while a low-confidence one gets `83 × 0.3 = 24.9`.

### 3.3 No SMOTE

Rationale from 2.2 — only 592 malicious samples across 5 heterogeneous attack types. Interpolating between DDoS and credential stuffing produces synthetic examples that represent no real attack pattern.

**Trade-off:** We accept lower recall on rare attack types (zero_day_exploit has the fewest samples) in exchange for not contaminating the minority class.

## 4. Primary Metric: PR-AUC

**Decision:** Optimize and report PR-AUC as the primary metric. F1 and FPR as secondary.

**Why not accuracy?**
- Dataset is 98.8% negative. A model that predicts "benign" for everything gets 98.8% accuracy.

**Why not ROC-AUC?**
- ROC-AUC includes TNR (true negative rate) which is always near 1.0 with extreme imbalance. This inflates the score and masks poor recall.
- PR-AUC focuses entirely on the minority class: how precise is the model when it says "malicious" (precision) and how many actual attacks does it catch (recall)?

**Why not F1 alone?**
- F1 is threshold-dependent — a single number at a single threshold. PR-AUC summarizes performance across all thresholds.
- We report F1 as secondary because the assessment asks for it, and it's easy to interpret.

**FPR:** Reported because the assessment explicitly asks for it and it connects to the $2.50 false positive cost in section 3.3.

## 5. Hyperparameter Tuning (Optuna)

**Decision:** Bayesian optimization with Optuna, 10-30 trials per model, using expanding-window temporal CV PR-AUC as objective.

**Why Optuna over grid search?**
- Bayesian search (TPE sampler) finds better hyperparameters with fewer trials than grid search.
- MedianPruner early-stops unpromising trials based on intermediate CV fold results — saves compute.
- With 50k rows, individual trials are fast (seconds), so 80 trials takes minutes, not hours.

**Why Optuna over random search?**
- Random search is a strong baseline but TPE converges faster, especially for correlated hyperparameters (e.g., learning_rate and n_estimators).

**Why not just sensible defaults?**
- Default hyperparameters leave significant performance on the table, especially for the class imbalance parameters (scale_pos_weight) and regularization.
- Ness et al. (2025) used default parameters and achieved 0.85 test accuracy. With Optuna, we should exceed that.

### Search Spaces

**Logistic Regression** (10 trials — fast, small space):
- `C`: [1e-4, 1e2] log scale — regularization strength
- `l1_ratio`: [0.0, 1.0] — elasticnet mixing (0=L2, 1=L1). Uses `penalty='elasticnet'` with `solver='saga'`. Note: sklearn 1.8+ deprecates explicit `penalty` selection; a future iteration should migrate to using `l1_ratio` and `C` directly without the `penalty` parameter.

**Random Forest** (20 trials):
- `n_estimators`: [100, 500]
- `max_depth`: [5, 20]
- `min_samples_leaf`: [5, 50] — prevents overfitting to small minority leaf nodes
- `max_features`: sqrt, log2 — feature subsampling per split

**XGBoost** (30 trials):
- `n_estimators`: [100, 800]
- `max_depth`: [3, 10]
- `learning_rate`: [0.01, 0.3] log scale
- `subsample`: [0.6, 1.0] — row subsampling
- `colsample_bytree`: [0.5, 1.0] — feature subsampling
- `min_child_weight`: [1, 20] — minimum samples per leaf
- `scale_pos_weight`: [50, 150] — class imbalance
- `reg_alpha`: [1e-8, 10] log — L1 regularization
- `reg_lambda`: [1e-8, 10] log — L2 regularization

**LightGBM** (30 trials):
- `n_estimators`: [100, 800]
- `num_leaves`: [15, 127] — leaf-wise specific
- `max_depth`: [3, 10]
- `learning_rate`: [0.01, 0.3] log scale
- `subsample`: [0.6, 1.0]
- `colsample_bytree`: [0.5, 1.0]
- `min_child_samples`: [5, 50]
- `scale_pos_weight`: [50, 150]
- `reg_alpha`: [1e-8, 10] log
- `reg_lambda`: [1e-8, 10] log

**Optuna impact on edge deployment:** Zero. Optuna runs offline during training only. The final model artifact is identical regardless of how hyperparameters were found.

## 6. Feature Importance & Post-Training Pruning

**Decision:** Extract feature importances from all models, then prune the best model and retrain.

### 6.1 Importance Extraction Methods

| Model | Method | Why |
|---|---|---|
| Logistic Regression | Absolute coefficient values | Coefficients directly measure feature contribution to log-odds. |
| Random Forest | Gini importance (`feature_importances_`) | Default scikit-learn impurity-based importance. Fast and sufficient for the 15-feature baseline. |
| XGBoost | `feature_importances_` (gain-based) | Gain measures total loss reduction from splits on each feature. More meaningful than split count, which favors high-cardinality features. |
| LightGBM | `feature_importances_` (gain-based) | Same reasoning as XGBoost. |

### 6.2 Pruning Process

1. Train best model with all 15 request-level features on the training set.
2. Rank features by gain importance.
3. Compute cumulative importance at multiple thresholds (90%, 95%, 99%) to compare aggressiveness.
4. Drop features below the chosen cutoff.
5. Retrain with pruned feature set using same hyperparameters.
6. Compare PR-AUC and per-attack recall on the held-out test set.

**Why 99% threshold?**
- 90% keeps only 4 features, 95% keeps 5, 99% keeps 8 out of 15.
- At 99%, the pruned model loses only -0.0006 PR-AUC (0.6817 → 0.6811) with identical F1 and per-attack recall — confirming the dropped 7 features were noise.
- The notebook reports all three thresholds (90%, 95%, 99%) for transparency.

### 6.3 Per-Attack-Type Breakdown

Using the `attack_class` column, report recall per attack type (credential_stuffing, ddos_l7, api_abuse, scanner, zero_day_exploit). This reveals which attacks rely on which features — if pruning drops a feature that was critical for zero_day_exploit detection, we'd see that attack type's recall collapse.

### 6.4 Connection to 2.2 Validation

Compare actual feature importance rankings with the predicted rankings from section 10 of `2.2-feature-engineering-decisions.md`. The baseline uses only the 15 request-level features (session features excluded due to data leakage — see 2.2 section 4). The tree model's feature importance reveals which request-level signals the model actually relies on for classification.

### 6.5 Connection to 2.4

The pruned model is what gets exported for edge deployment. Fewer features = fewer tree splits = smaller WASM binary = faster inference. The pruning analysis in 2.3 directly feeds into the deployment feasibility analysis in 2.4.

## 7. Threshold Analysis

**Decision:** Report metrics at default threshold (0.5) AND find cost-optimal threshold using the assessment's cost structure.

### 7.1 Default Threshold (0.5)

Standard comparison table for all 4 models (temporal split, 15 request-level features):

| Model | CV PR-AUC | PR-AUC | ROC-AUC | Precision | Recall | F1 | FPR |
|---|---|---|---|---|---|---|---|
| Logistic Regression | 0.4945 | 0.6354 | 0.9042 | 0.7947 | 0.6163 | 0.6943 | 0.0019 |
| Random Forest | 0.5017 | 0.6161 | 0.9508 | 0.3595 | 0.6163 | 0.4541 | 0.0129 |
| XGBoost | 0.5215 | 0.6817 | 0.9572 | 0.3612 | 0.6163 | 0.4555 | 0.0128 |
| LightGBM | 0.5144 | 0.6668 | 0.9546 | 0.4748 | 0.6163 | 0.5364 | 0.0080 |

### 7.2 Cost-Optimal Threshold

Using the cost structure from section 3.3 of the assessment:
- FP cost (blocking legitimate request): $2.50
- FN cost (attack reaching origin): $0.10

**Expected cost per request at threshold t:**

```
Cost(t) = FPR(t) × (1 - π) × $2.50  +  FNR(t) × π × $0.10
```

Where π = attack base rate.

- Dataset base rate: ~1.2% (592/50000). Production base rate: 0.05%.
- Compute cost curve across thresholds [0.01, 0.99] for best model (XGBoost).
- Find t* minimizing total expected cost.

**Results:**

| Scenario | Threshold | Precision | Recall | F1 | FPR |
|---|---|---|---|---|---|
| Default | 0.500 | 0.3612 | 0.6163 | 0.4555 | 0.012776 |
| Normal costs ($2.50/$0.10) | 0.964 | 1.0000 | 0.6163 | 0.7626 | 0.000000 |
| High FN costs ($2.50/$5.00) | 0.964 | 1.0000 | 0.6163 | 0.7626 | 0.000000 |

Both cost scenarios converge to the same threshold (0.964) because the model's probability distribution is bimodal — high-confidence predictions cluster near 0 or near 1, with few predictions in the 0.5–0.96 range. Raising the threshold from 0.5 to 0.964 eliminates all false positives without reducing recall, indicating the model's uncertain predictions (0.5–0.96) were predominantly false positives.

**Why do this in 2.3?**
- The assessment asks for precision/recall/F1/FPR — these depend on threshold choice.
- Reporting only at 0.5 is naive for extreme imbalance.
- Shows awareness of the precision-recall trade-off. Section 3.3 then deepens the math.

## 9. Expected vs Actual Outcomes

Based on the 15 request-level features and literature:

- **Tree models outperform LR in ranking (PR-AUC)** — Confirmed. XGB PR-AUC 0.6817 vs LR 0.6354. However, the gap is smaller than expected — with only 15 request-level features, the non-linear signal is limited. Notably, LR achieves the highest precision (0.7947) at default threshold due to its conservative decision boundary.
- **XGBoost and LightGBM are close** — Confirmed. XGB 0.6817, LGBM 0.6668 PR-AUC. XGB slightly edges out, likely due to better handling of the extreme imbalance via tuned `scale_pos_weight`.
- **Feature pruning at 99% is nearly lossless** — Confirmed. 15 → 8 features at 99% cumulative gain. Pruned model PR-AUC: 0.6811 (-0.0006), identical F1 and per-attack recall. The 7 dropped features (`path_has_params`, `ua_entropy`, `ua_is_browser`, `is_sensitive_endpoint`, `has_accept_language`, `has_referer`, `has_cookie`) contribute <1% of total gain.
- **Cost-optimal threshold shifts from 0.5** — Confirmed. Both cost scenarios (normal and high FN) converge to threshold 0.964, achieving perfect precision (1.0) with recall 0.6163 and zero FPR. Default 0.5 gives 0.36 precision — 2 out of 3 alerts are false positives.
- **zero_day_exploit is undetectable** — Confirmed. Recall 0.0 across ALL models. With 0 training samples for this attack type (it only appears in test set days 10-12), no supervised model can learn to detect it. This is an honest limitation and a strong argument for the unsupervised/hybrid approach discussed in Part 3.2.
- **ddos_l7 has poor recall without session features** — Confirmed. Only 0.1287 recall across all models. Individual DDoS requests look identical to normal traffic — volumetric attacks can only be detected through session-level aggregation (request rate, timing regularity).

## 10. Data Leakage Lessons Learned

During development, three sources of data leakage were discovered and addressed:

1. **Server-response features (3 features excluded):** `status_code_group`, `response_time_ms`, and `body_size_bytes` are server-side response attributes that reflect how the server processed the request. At inference time, the model must classify the request *before* the server responds. Including them produced PR-AUC 1.0 on all tree models because status codes (403/429) directly encode the server's existing malicious-traffic decisions. These were moved to `META_COLUMNS` in `src/model.py`.

2. **Session-level features (78 features excluded):** `compute_session_features()` in `src/features.py` uses `groupby().transform()` over the full dataset. Each request's session aggregates include information from future requests and test-set rows. Fix requires causal windowing — planned for future iteration.

3. **Frequency encodings (4 features excluded):** `compute_frequency_encoding()` computes value frequencies (`method_freq`, `country_freq`, `asn_freq`, `tls_fingerprint_freq`) over the full dataset, leaking test distribution into training features. Fix requires fitting on training data only.

The baseline model uses only the 15 non-leaky request-level features. This is documented as a known limitation — the session features are valuable in production (where they're computed in real-time from streaming counters), but the offline training pipeline needs causal windowing before they can be safely used.
