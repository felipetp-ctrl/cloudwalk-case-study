# 2.2 Feature Engineering — Decision Dump

This document captures every design choice, trade-off, and justification for the feature engineering pipeline. It serves as the reasoning backbone for the implementation and writeup.

---

## 1. Aggregation Granularity

**Decision:** Compute session-level features at **two parallel granularities** — by `source_ip` AND by `tls_fingerprint` — independently.

**Why not just one?**
- IP-only misses IP-rotating attackers (common in credential stuffing botnets).
- TLS-only creates false positives because legitimate browsers with the same version share JA3 hashes.
- Both together let the model disambiguate: "rare TLS + high IP volume = attack" vs "common TLS + high IP volume = flash sale behind NAT."

**Trade-off:** 2x the state storage at inference time (~2-3KB per active source). Acceptable given the 50k req/s throughput constraint — this is counter lookups, not heavy computation.

**Alternative considered:** Composite key (IP + TLS). Rejected because it creates sparser groups with unreliable aggregates and cold-start problems for new IP/TLS combinations.

---

## 2. Time Windows

**Decision:** Three fixed windows — **1 minute, 5 minutes, 30 minutes**.

**Why three?**
- 1min captures burst attacks (L7 DDoS).
- 5min captures medium-speed campaigns (API abuse, scanner sweeps).
- 30min captures low-and-slow attacks (credential stuffing that spaces requests to evade rate limiters).

**Why fixed, not sliding?**
- Fixed windows are trivially pre-computable for training (just group-by on truncated timestamps).
- At inference, fixed windows map to simple TTL-based counters in a state store — no need to maintain sorted request histories.
- Sliding windows provide marginally better signal but require maintaining per-source request logs, which is expensive at 50k req/s.

**Trade-off:** Fixed windows split attacks that straddle boundaries. Acceptable because having three overlapping granularities means at least one window captures the bulk of any attack pattern.

**Alternative considered:** Single 5min window. Rejected — misses both fast bursts and slow campaigns. The 3x feature count is not a problem for tree models.

## 3. Per-Request Features (15 active, 3 excluded)

### 3.1 From `http_requests`

| Feature | Type | Rationale |
|---|---|---|
| `path_depth` | int | Count of `/` segments. Deep paths indicate directory traversal or API enumeration. |
| `path_length` | int | Character count of path. Abnormally long paths suggest fuzzing/injection attempts. |
| `path_entropy` | float | Shannon entropy of path characters. High entropy = randomized/obfuscated paths (scanners, fuzzers). |
| `path_has_params` | bool | Whether path contains `?` or `=`. Query-string patterns differ between normal and scanning traffic. |
| `ua_length` | int | User-agent string length. Very short or very long = likely non-browser. |
| `ua_entropy` | float | Shannon entropy of user-agent characters. Randomized UAs differ from real browser strings. |
| `ua_is_browser` | bool | Regex for known browser signatures (Mozilla, Chrome, Safari, Firefox). |
| `ua_is_bot_library` | bool | Regex for python-requests, curl, wget, Go-http-client, etc. |
| `hour_of_day` | int | Extracted from timestamp. Attack campaigns often cluster in off-hours. |
| `is_sensitive_endpoint` | bool | Path matches `/auth`, `/login`, `/payment`, `/tokenize` patterns. Attacks concentrate on high-value endpoints. |

### 3.2 From `request_headers`

| Feature | Type | Rationale |
|---|---|---|
| `header_count` | int | Total headers per request. Bots send fewer headers than real browsers (typically 8-15 vs 3-5). |
| `has_accept_language` | bool | Browsers almost always send this; bots often omit it. |
| `has_referer` | bool | Missing Referer on navigation-like requests = suspicious. |
| `has_cookie` | bool | Absence on authenticated endpoints signals automated tools. |
| `has_authorization` | bool | Presence on non-auth endpoints or absence on protected ones is anomalous. |

### 3.3 Decisions on per-request features

**Server-response features (removed from baseline):** Three features from `http_requests` — `status_code_group` (bucketed status code), `response_time_ms`, and `body_size_bytes` — were initially included but removed after discovering they constitute **data leakage**. These are server-side response attributes, not request attributes: the status code reflects whether the server already identified/blocked the request, response time reflects server-side processing, and body size reflects the response payload. At inference time, the model must classify the request *before* the server processes it, so these fields are unavailable. Including them produced PR-AUC 1.0 on all tree models — the model was trivially learning "4xx status = malicious."

**Frequency encoding (removed from baseline):** Initially included `method_freq`, `country_freq`, `asn_freq`, and `tls_fingerprint_freq` as frequency-encoded categoricals. These were removed from the baseline model because frequency encoding computed over the full dataset introduces **data leakage** — the frequency of an ASN or country in the test set leaks information about the test distribution into training. Proper frequency encoding requires fitting on training data only and transforming test data separately. This can be reintroduced in a future iteration with proper train-only fitting via `sklearn.preprocessing` pipelines.

**Entropy features:** Included for `path` and `user_agent` because they capture randomization/obfuscation signals that no other feature covers. Cheap to compute (O(n) character scan). Particularly useful for scanner detection where paths are generated programmatically.

**Header approach:** Count + key presence flags only. Rejected full header fingerprinting (ordered header-name hash) because:
- Adds a high-cardinality categorical needing special encoding.
- TLS fingerprint already captures tool identity at a lower level.
- Marginal signal gain doesn't justify the complexity for edge deployment.

## 4. Session/Source-Level Aggregated Features (78 total) — EXCLUDED FROM BASELINE

> **Status:** Designed and implemented (`src/features.py:compute_session_features`), but **excluded from the 2.3 baseline model** due to data leakage. The current implementation uses `groupby` + `transform` over the full dataset, meaning each request's session features include information from future requests and from test-set rows. This produces artificially high model performance (PR-AUC 1.0 on all tree models).
>
> **Fix required:** Causal windowing — session features must only aggregate past requests within each time window. This requires either (a) an online/streaming computation that processes rows in timestamp order, or (b) a rolling-window join that excludes the current and future rows. This is planned for a future iteration.
>
> **The baseline model (2.3) uses only the 15 non-leaky request-level features.** The session feature design below remains valid for the production architecture, where features are computed incrementally from a real-time state store (counters updated per-request, never looking ahead).

Computed at 2 granularities (IP, TLS) x 3 windows (1m, 5m, 30m) = 6 variants per base feature.
Prefix convention: `{ip|tls}_{1m|5m|30m}_{feature_name}`

| Base Feature | Type | Rationale |
|---|---|---|
| `request_count` | int | Volume signal. DDoS = high at 1m; credential stuffing = sustained at 30m. Expected to be the single strongest session-level predictor. |
| `unique_paths` | int | Endpoint diversity. Scanners enumerate many paths; normal users stick to a few flows. |
| `path_entropy` | float | Shannon entropy over distinct paths visited. Complements unique_paths. |
| `error_rate` | float | Ratio of 4xx+5xx responses. Brute-force and stuffing generate high error rates (401/403 spikes). |
| `unique_status_codes` | int | Distinct status codes received. Normal sessions see 200/304; attacks trigger diverse errors. |
| `avg_response_time` | float | Mean response time in window. Resource-exhaustion attacks skew this upward. |
| `std_response_time` | float | Response time variance. Very low std = automated regularity; very high = probing heterogeneous endpoints. |
| `avg_body_size` | float | Mean body size. Replay attacks send identical payloads; scanners send zero-body probes. |
| `std_body_size` | float | Body size variance. Near-zero std = replayed/templated payloads (strong automation signal). |
| `method_diversity` | int | Unique HTTP methods used. Normal users: GET+POST; scanners add OPTIONS, PUT, DELETE. |
| `inter_request_time_mean` | float | Average time between consecutive requests. Bots are unnaturally fast or metronomically regular. |
| `inter_request_time_std` | float | Timing variance. Very low std = bot-like regularity. |
| `sensitive_endpoint_ratio` | float | Fraction of requests to auth/payment/tokenize paths. Credential stuffing hammers login; normal users have mixed traffic. |

**Total: 13 base x 6 variants = 78 session-level features (excluded from baseline, planned for future iteration).**

## 5. Why NOT PCA / Dimensionality Reduction

**Decision:** No PCA or other linear dimensionality reduction.

**Reasons:**
1. **Tree models don't benefit.** LightGBM/XGBoost split on individual features and handle correlated/high-dimensional data natively with implicit feature selection. PCA helps linear models and neural nets.
2. **Destroys interpretability.** "PC17 is important" is meaningless. "ip_5m_error_rate is the top predictor" is actionable for the security team and required by the assessment's justification requirement.
3. **Assumes linearity.** The non-linear interactions between features (request_count × error_rate, timing regularity × endpoint diversity) are exactly what tree splits exploit. PCA would compress these into components that obscure the signal.

**Instead: post-training feature importance pruning.**
- Train with all 15 request-level features (baseline) or full feature set once session features are fixed.
- Use LightGBM's `feature_importance(importance_type='gain')` to rank features.
- Drop the bottom 20-30% that contribute negligible gain.
- Retrain final model with the surviving feature set.
- This reduces model size for WASM deployment (fewer splits = smaller tree) while preserving interpretability.

## 6. Confidence Levels as Sample Weights

**Decision:** Use `incident_labels.confidence` as **sample weights during training**, not as input features.

**Why weights, not features?**
- Confidence reflects label quality, not request characteristics. A request doesn't "look" high-confidence — its label is high-confidence because the security team is certain about the incident.
- As a feature, it would leak labeling metadata into the model. At inference, you don't have confidence levels for new requests.
- As weights, high-confidence labels (521 of 592 malicious) influence the loss function more, so the model trusts cleaner labels more during training.

**Weight mapping:** `high=1.0`, `medium=0.6`, `low=0.3` (tunable). Combined with `scale_pos_weight` for class imbalance.

---

## 7. Class Imbalance Strategy

**Decision:** `scale_pos_weight` (XGBoost) / `class_weight='balanced'` (LightGBM) + confidence-based sample weights. No SMOTE.

**Why not SMOTE?**
- Only 592 malicious samples across 5 heterogeneous attack types (credential_stuffing, ddos_l7, api_abuse, scanner, zero_day_exploit).
- SMOTE interpolates between nearest neighbors — interpolating between a DDoS sample and a credential stuffing sample produces a synthetic example that represents no real attack pattern.
- With heterogeneous positives, synthetic data contaminates the minority class.

**Why class weights instead?**
- Penalizes errors on the minority class proportionally to the imbalance ratio (~83:1 in our dataset, though production is ~1000:1).
- Achieves the same effect as oversampling without fabricating data.
- Combined with confidence weights, the model focuses most on high-confidence minority samples.

**Focal loss considered:** Useful if the model has many "easy" high-confidence benign predictions dominating the gradient. Worth experimenting with, but class weights are the baseline. Focal loss adds a hyperparameter (gamma) and may not help if the tree model already handles easy cases well through depth-limited splits.

**Primary metric:** Precision-Recall AUC, not accuracy (useless at 98.8% negatives). F1 as secondary. Special attention to FPR given the $2.50 cost per false positive (see assessment 3.3).

## 8. Feature Lifecycle in Production

**Connection to 2.2:** Feature design choices must account for how features are maintained, monitored, and updated in production. A feature that's perfect today but unmaintainable is a bad feature.

### 8.1 Retraining Cadence

- **Scheduled weekly retraining** aligned with the assessment's "concept drift changes weekly" constraint.
- **Emergency retraining** triggered by drift alerts between cycles.
- Each retraining cycle: new labels (1-3 day delay) joined to recent traffic → features recomputed → model retrained → feature importance re-evaluated.

### 8.2 Feature Importance Re-evaluation

- Part of every retraining run, not a separate process.
- Features that were important last week may be useless this week (attackers adapted) and vice versa.
- Example: if attackers start randomizing JA3 per-request (scenario 3.4), `tls_*` features lose importance → model drops them and relies more on behavioral features (timing, path patterns, error rates).

### 8.3 Feature Drift Monitoring

Tracked per-feature between retraining cycles:
- **PSI (Population Stability Index)** or **KS-test** on each feature's distribution vs training baseline.
- **Prediction confidence distribution** — if the model becomes less confident overall, feature distributions may have shifted.
- Threshold breaches trigger alerts → emergency retraining.

### 8.4 Feature Set Versioning

- Each model version ships with a **feature manifest** — the exact list of features, their computation logic, and expected distributions.
- Deployment is a model swap at the edge: push new WASM binary with updated model + feature config.
- State store counters keep running across model versions; only the model consuming them changes.

### 8.5 Implication for Feature Design

This lifecycle informed two design choices:
1. **All features are simple aggregates** (counts, means, ratios, entropy). No features require complex stateful logic that would break across model version swaps.
2. **No PCA or learned transformations.** Raw interpretable features can be independently monitored for drift. A PCA component drifting tells you nothing about what changed in the underlying data.

## 10. Expected High-Predictive-Power Features

### Baseline (request-level only)

Ranked by expected importance for the 15 request-level features:

1. **`ua_is_bot_library`** — Direct signal, though sophisticated attackers spoof this.
2. **`header_count`** — Simple but effective browser vs bot discriminator.
3. **`has_authorization`** — Presence/absence on specific endpoints reveals automation patterns.
4. **`path_depth` / `path_entropy`** — Catches fuzzers and automated path generation.
5. **`hour_of_day`** — Attack campaigns often cluster in off-hours.
6. **`ua_is_browser` / `ua_length`** — Browser vs bot user-agent fingerprinting.
7. **`is_sensitive_endpoint`** — Attack concentration on auth/payment endpoints.
8. **`has_cookie` / `has_accept_language`** — Browser vs bot fingerprinting via header presence.

### Future (with session features)

Once session features are fixed with causal windowing, these are expected to dominate:

1. **`ip_1m_request_count`** — Volume is the most direct DDoS/brute-force signal.
2. **`ip_5m_error_rate`** — Credential stuffing generates distinctive 401/403 error patterns.
3. **`tls_5m_inter_request_time_std`** — Bot regularity vs human irregularity.
4. **`ip_5m_sensitive_endpoint_ratio`** — Attacks concentrate on auth/payment paths.

## 11. Empirical Validation Results

> **Note:** The original empirical validation (section removed) was conducted on the full 100-feature set including session features computed with data leakage. Those results showed artificially high discrimination ratios (e.g., `tls_30m_inter_request_time_std` at 209x) that were artifacts of lookahead bias, not genuine signal. Updated empirical results from the 18-feature baseline model are documented in `2.3-baseline-model-decisions.md`.

### Data Leakage Discovery

During baseline model training (2.3), three sources of data leakage were discovered and addressed:

1. **Server-response features:** `status_code_group`, `response_time_ms`, and `body_size_bytes` are server-side response attributes — they reflect how the server processed/blocked the request, not the request itself. At inference time, the model must classify the request *before* the server responds, so these fields are unavailable. Including them produced PR-AUC 1.0 on all tree models because status codes like 403/429 directly encode the server's own malicious-traffic decision.

2. **Session features:** `compute_session_features()` uses `groupby().transform()` over the full dataset, meaning each request's session-level aggregates include information from future requests and test-set rows. At inference time, session features would be computed incrementally from a state store, never seeing the future.

3. **Frequency encodings:** `compute_frequency_encoding()` computes value frequencies over the entire dataset. Test-set frequencies leak into training features. Proper implementation requires fitting on training data only.

All three issues were addressed by excluding these features from the baseline model. The session feature architecture remains sound for production (where features are computed in real-time from streaming counters), but the offline training pipeline needs causal windowing before they can be used for model training.

### Performance Note

Per-request feature computation completes in ~2 seconds on 50k rows. At the edge, these features are computed inline per-request in microseconds — no state store required.
