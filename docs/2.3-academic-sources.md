## Source 1: Ness et al. (2025) — "Anomaly Detection in Network Traffic Using Advanced Machine Learning Techniques"

**Citation:** S. Ness, V. Eswarakrishnan, H. Sridharan, V. Shinde, N. V. P. Janapareddy, and V. Dhanawat, "Anomaly Detection in Network Traffic Using Advanced Machine Learning Techniques," *IEEE Access*, vol. 13, pp. 16133–16145, Jan. 2025. DOI: 10.1109/ACCESS.2025.3526988

**Dataset:** KDDCup'99 (network-layer traffic, not HTTP L7)

**Key findings:**
- Compared Isolation Forest, Naive Bayes, XGBoost, LightGBM, SVM
- **LightGBM:** Training accuracy 1.0, test accuracy 0.85. Learned complex patterns with reasonable generalization.
- **XGBoost:** Training accuracy 0.99, test accuracy 0.83. Strong but slightly below LightGBM.
- **SVM:** Training ~0.99, test ~0.85. Strong baseline but diminishing returns at high accuracy.
- **Naive Bayes:** Training 0.89, test 0.81. Decent but misses complex feature interactions.
- **Isolation Forest:** Training 0.5, test 0.4. Poor — anomaly-based approach struggled with multi-class complexity.

**Relevance to our project:**
- Validates LightGBM and XGBoost as top performers for network anomaly detection
- Supports our decision to use gradient boosted trees over unsupervised methods
- KDDCup'99 is network-layer (TCP/IP features), ours is L7 HTTP — different feature space but same model selection logic

**Limitations:**
- Used accuracy as primary metric, not PR-AUC — problematic for imbalanced data
- No temporal train/test split or concept drift discussion
- Default hyperparameters, no tuning

## Source 2: Kamol et al. (2024) — "Credentials Stuffing Attack Prevention Using Machine Learning"

**Citation:** M. M. Kamol, M. S. Siddiky, F. Anwar, A.-M. Khan, and A. Salam, "Credentials Stuffing Attack Prevention Using Machine Learning," in *Proc. 27th Int. Conf. Computer and Information Technology (ICCIT)*, Cox's Bazar, Bangladesh, Dec. 2024, pp. 2899–2903. DOI: 10.1109/ICCIT64024.11022023

**Dataset:** Synthetic web access logs (40,374 entries: 20,239 attack, 20,135 benign)

**Key findings:**
- Compared clustering algorithms: K-means, DBSCAN, Gaussian Mixture, BIRCH, OPTICS, Hierarchical
- **DBSCAN achieved 90.56% accuracy** — best performer by large margin
- K-means: 49.84%, BIRCH: 0.24%, others ~50%
- Feature approach: aggregated by IP address, encoded URL visit patterns into matrix
- Analyzed real credential stuffing tools (Sentry MBA, SNIPR, OpenBullet) for attack patterns

**Relevance to our project:**
- Validates IP-level aggregation for credential stuffing detection
- Supports our session-level feature design (grouping by source IP/fingerprint)
- Our supervised approach should outperform given we have labels
- Confirms behavioral patterns (access sequences, timing) distinguish attackers

**Limitations:**
- Balanced dataset (50/50) — doesn't address extreme class imbalance
- Synthetic data, simplified attack patterns
- Unsupervised only, no temporal validation

## Source 3: Dawadi et al. (2023) — "Deep Learning Technique-Enabled Web Application Firewall for the Detection of Web Attacks"

**Citation:** B. R. Dawadi, B. Adhikari, and D. K. Srivastava, "Deep Learning Technique-Enabled Web Application Firewall for the Detection of Web Attacks," *Sensors*, vol. 23, no. 4, p. 2073, Feb. 2023. DOI: 10.3390/s23042073

**Dataset:** Simulation data for DDoS, XSS, SQL injection

**Key findings:**
- Layered WAF architecture: Layer 1 (DDoS) → Layer 2 (XSS/SQLi)
- **LSTM DDoS detection: 97.57% accuracy**
- **LSTM XSS/SQLi: 89.34% accuracy**
- Layered approach filters high-volume DDoS first, reducing load on Layer 2
- Signature-based WAFs fail against zero-day attacks; ML generalizes better

**Relevance to our project:**
- **Contrast case:** LSTM achieves high accuracy but requires sequential processing and more compute — incompatible with <5ms WASM constraint
- Justifies choosing trees over deep learning: comparable detection, orders-of-magnitude less inference cost
- Layered architecture concept could inform future work
- Confirms WAF as correct deployment point for HTTP traffic

**Limitations:**
- No latency/throughput benchmarks
- LSTM needs GPU or significant CPU
- Single-dataset evaluation

## Source 4: Mayhew et al. (2015) — "Use of Machine Learning in Big Data Analytics for Insider Threat Detection"

**Citation:** M. Mayhew, M. Atighetchi, A. Adler, and R. Greenstadt, "Use of Machine Learning in Big Data Analytics for Insider Threat Detection," in *Proc. IEEE Military Communications Conf. (MILCOM)*, Tampa, FL, USA, Oct. 2015, pp. 915–920. DOI: 10.1109/MILCOM.2015.7357562

**System:** BBAC (Behavior-Based Access Control) — US Air Force Research Lab

**Key findings:**
- Pipeline: Feature Extraction → Feature Enrichment → K-Means Clustering → SVM Classification → Decision Tree interpretation
- HTTP-specific features: target URL, time of day, requester host ID, header analysis
- Feature enrichment computes aggregate statistics (periodicity, connection counts, traffic volumes) — mirrors our session-level features
- K-Means clusters entities by behavior profile, then trains per-cluster SVMs
- Decision trees for interpretability
- **Real-time classification latency: ~10ms** (tens of milliseconds)
- SVM classification latency "in the microsecond range on modern computers"

**Relevance to our project:**
- Validates feature engineering approach: per-request extraction + session-level aggregation
- Behavioral profiling (learning "normal" per entity) aligns with our IP/TLS-level aggregation
- Latency ~10ms is comparable to our <5ms constraint — tree-based models can meet real-time requirements
- Cluster-then-classify is interesting but adds complexity vs our single-model approach

**Limitations:**
- Insider threats, not external attacks
- 2015 paper — predates modern gradient boosting
- No concept drift or retraining discussion

## Cross-Cutting Themes

### Model Selection Consensus
All sources comparing models found gradient boosted trees (XGBoost, LightGBM) or SVMs as top performers. Deep learning achieves slightly higher accuracy at significantly higher computational cost — favors trees for edge deployment.

### Feature Aggregation Validation
Kamol et al. and Mayhew et al. independently validate aggregating request-level data by source (IP/fingerprint) and computing behavioral features. This is the core of our 2.2 feature engineering.

### Gaps Our Work Addresses
1. **Class imbalance:** No source rigorously handles <0.1% positive rate. Ness uses accuracy on moderate imbalance. Kamol uses balanced data.
2. **Temporal validation:** No source uses temporal splits — all use random CV.
3. **Edge deployment:** Only Mayhew discusses real-time latency (~10ms). None address WASM or model size.
4. **Cost-aware thresholds:** No source considers the economic trade-off between FP and FN costs.
